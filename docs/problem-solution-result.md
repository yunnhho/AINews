>
> - **Part 1** — 설계 단계에서 **예상한 문제**와 선제 대응 (기술 의사결정)
> - **Part 2** — 개발·검증 중 **실제로 일어난 문제**와 수정 (버그 기록)

>
> **수치 표기 규칙**
> - `[측정]` : 코드/벤치마크/실행으로 실제 확인한 값
> - `[설계목표]` : 임계값·파라미터 등 구현에 반영된 설계 기준값

# Part 1 — 예상한 문제와 선제 대응 (기술 의사결정)

## 1-1. 한국어 검색 품질 — Meilisearch → Elasticsearch(nori)

**문제 (예상 → 실측으로 확인)**
- 초기 검색 엔진으로 가벼운 Meilisearch를 도입했으나, 한국어 형태소 단위 검색이 약했다.
- `"RAG 패턴"` 검색 시 `"RAG"`/`"패턴"`이 조사·어미와 분리되지 않아 연관 문서 누락 발생.
  `임베딩을`/`임베딩이` 같은 조사 변형이 서로 다른 토큰으로 인식돼 재현율 저하.
- 카드 요약·문제·아이디어 필드가 **한국어 번역문**이라 검색 품질이 곧 서비스 품질.

**해결**
- **Elasticsearch 8.17 + nori 형태소 분석기**로 전환 (P6-2 / Session 8).
- `nori_tokenizer` + `nori_part_of_speech` 필터로 커스텀 `korean` analyzer 구성,
  nori 플러그인 포함 ES 이미지 직접 빌드(`backend/es/Dockerfile`).
- 검색 필드(`title`/`summary`/`problem`/`idea`)는 `korean` analyzer + `.en`(english) 서브필드,
  필터 필드(`category`/`card_type`/`difficulty`/`tags`)는 `keyword`로 분리 매핑.
- 카드 INSERT 시 자동 인덱싱(`tasks/index_card.py`), `es/setup.py`로 멱등 초기화 + 전체 재색인.

```
backend/app/services/search.py   — AsyncElasticsearch 래퍼 + nori 매핑
backend/es/mappings/cards.json   — 매핑 단일 출처
backend/es/setup.py              — 인덱스 생성 + 전체 재인덱싱
```

**결과**

| 항목 | 이전(Meilisearch) | 이후(ES + nori) |
|---|---|---|
| 한국어 형태소 분석 | 미지원(공백/단순 토큰) | nori 형태소 단위 색인 `[설계목표]` |
| 복합 필터(카테고리+타입+난이도) | 부분 지원 | term 필터 조합 완전 지원 `[측정]` |
| `"RAG"` → `"RAG 패턴"` 연관 매칭 | 누락 | 형태소 매칭으로 재현 `[설계목표]` |
| 영어/키워드 검색 | — | `.en` 서브필드 + `tags.text`로 지원 `[측정]` |

## 1-2. LLM 번역 환각 — 역번역 코사인 검증 게이트

**문제 (예상)**
- 영문 원문을 Claude로 한국어 요약·번역하는데, LLM 번역은 **의미 왜곡·환각**이
  간헐적으로 발생할 수 있고(예: "context window"를 일반어로 의역), 사람이 전수 검수하면
  배치 파이프라인의 의미가 없어진다.
- 잘못된 번역이 그대로 발행되면 서비스 신뢰도에 직결 → 자동 품질 게이트 필요.

**해결**
- **역번역(back-translation) + 코사인 유사도** 검증 도입.
- KO 요약을 Claude로 다시 EN 역번역 → 원문과 `sentence-transformers`
  (`paraphrase-multilingual-MiniLM-L12-v2`) 임베딩 코사인 유사도 측정.
- **임계값 0.85 미만이면 발행 차단** → 최대 3회 재번역, 실패 시 `manual_review` 큐로
  이동해 Admin에서 수동 검토. 미통과 카드는 `is_published=False` 초안 저장.
- 검증 모델은 매 카드 재로딩하지 않도록 **워커 시작 시 사전 로딩**.

```
pipeline/pipeline/ai/backtranslate.py  — _SIM_THRESHOLD = 0.85
pipeline/pipeline/ai/publisher.py      — 미통과 카드 초안 처리
```

**결과**

| 항목 | 이전(검증 없음 가정) | 이후(역번역 게이트) |
|---|---|---|
| 번역 품질 보증 | 무검증 발행 | 코사인 ≥ 0.85만 자동 발행 `[설계목표]` |
| 저품질 번역 처리 | 사용자 노출 | manual_review 격리 `[측정]` |
| 운영 가시성 | 없음 | Admin 번역 통과율 KPI + 통과율 < 70% 경보 `[설계목표]` |

- 첫 실배치 5건 테스트: 2건 통과(0.887/0.928), 3건 검토 큐(0.70~0.79) — 게이트 실작동 `[측정]`.
- 한계 사례와 수동 검토 큐의 가치는 § 2-11 참조.

## 1-3. 멀티소스 중복 카드 — 2단계 디듀프

**문제 (예상)**
- 소스 그룹 A(RSS)·B(GitHub)·C(블로그)·D(뉴스레터) 동시 수집 →
  **같은 뉴스가 여러 소스에서 중복 유입** (예: 동일 프레임워크 릴리스를 RSS·블로그·뉴스레터가 모두 다룸).
- 단순 URL 비교만으로는 제목만 같고 URL이 다른 재게시·미러를 못 거른다.

**해결**
- 2단계 디듀프(`pipeline/filters/dedup.py`):
  1. **URL SHA-256 해시** 완전 일치 제거 (배치 내 + DB 기존 카드).
  2. **TF-IDF 타이틀 유사도** — `char_wb` 분석기 `ngram(2,3)`로
     코사인 유사도 **≥ 0.85** 쌍 제거. 충돌 시 **content가 더 긴 카드 유지**.
- 디듀프를 AI 처리 **이전 단계**에 배치해 중복분 토큰 낭비 원천 차단.

**결과**

| 항목 | 이전(URL만) | 이후(URL + TF-IDF) |
|---|---|---|
| 동일 URL 중복 | 제거 | 제거 `[측정]` |
| 제목 유사 재게시 | 통과(중복 노출) | 0.85 유사도로 제거 `[설계목표]` |
| 중복 시 보존 기준 | 임의 | 더 긴 본문 우선 `[측정]` |
| AI 비용 영향 | 중복분까지 호출 | 중복 제거 후 호출(비용 ↓) `[예상]` |

- 첫 실배치: RSS 609 + GitHub 8 수집 → 디듀프 후 562건 `[측정]`.
- char n-gram 기반이라 한국어/영어 제목 모두에서 띄어쓰기·조사 차이에 강건.

## 1-4. AI 호출 비용 폭증 — 프롬프트 분리 + 입력 절단 + 선(先)디듀프

**문제 (예상)**
- Claude API 호출이 파이프라인 최대 비용원. 무절제한 호출은 비용 폭증.
- 긴 원문 전체를 프롬프트에 넣으면 입력 토큰이 선형 증가.

**해결**
- **카드 타입별 프롬프트 분리**(NEWS / TECHNIQUE) — 불필요한 지시 토큰 제거.
- 원문은 **`content[:3000]` 절단** 후 요약 → 입력 토큰 상한 고정.
- 디듀프를 AI 단계 **앞단**에 배치(§1-3) → 중복분 호출 원천 차단.
- 코드 스니펫은 LLM 생성 금지, **원본에서 추출만** → 생성 토큰 절감 + 정확성.
- Admin 비용 대시보드(`/admin/costs/daily`)로 일별 토큰·USD + 예산 게이지 노출.
- (운영 중 보강) **월 예산 하드 캡** — 표시용에 그치던 예산을 실제 강제로 전환, § 2-14 참조.

**결과**

| 항목 | 이전(소박한 방식) | 이후(분리+절단+선디듀프+캡) |
|---|---|---|
| 입력 토큰/카드 | 원문 전체 비례 | 최대 3,000자 상한 `[설계목표]` |
| 중복 카드 AI 호출 | 발생 | 디듀프 후 0 `[측정]` |
| 코드 스니펫 토큰 | 생성 비용 발생 | 추출(생성 0) `[측정]` |
| 월 비용 상한 | 없음 | 하드 캡 $20 도달 시 AI 루프 중단 `[측정]` |
| 비용 가시성 | 없음 | 일별 KPI + 예산 게이지 `[측정]` |

- 실측 비용: 5건 테스트 배치 $0.0122, 카드 1건(B-4 CLAUDE.md) $0.0037 `[측정]`.

## 1-5. 외부 소스 불안정 — 헬스 추적 + 자동 비활성화 + 경보

**문제 (예상)**
- 외부 RSS/스크래핑/IMAP 소스는 **언제든 죽거나 구조가 바뀐다**.
- 죽은 소스를 매 배치 재시도하면 자원 낭비 + 장애 인지 지연.
- 개별 소스 장애(GitHub 403, 블로그 다운)가 배치 전체 실패로 번질 위험.

**해결**
- `source_health`로 소스별 연속 실패 추적. 어댑터 개별 실패는 격리하고
  나머지 소스 수집은 계속 진행.
- **연속 실패 3회 이상 → Slack + 이메일 즉시 경보** (동일 소스 1시간 쿨다운).
- 404 등 영구 실패는 **자동 비활성화**(`enabled=False`)로 재시도 차단.
- 배치 전체 수집 0건도 즉시 경보. 배치 실패 시 15분 후 자동 재시도(`max_retries=2`).
- 장시간 가동 워커의 메모리 누적 대비 `--max-tasks-per-child=50`으로 프로세스 주기 재생성.

```
pipeline/models/source_health.py       — 실패 카운트 + 자동 비활성화
backend/app/services/alerting.py       — Slack/SMTP + 쿨다운
pipeline/tasks/check_source_health.py  — 배치 후 상태 점검
```

**결과**

| 항목 | 이전(무감시 가정) | 이후(헬스 + 경보) |
|---|---|---|
| 장애 인지 | 수동 발견 | 연속 3회 실패 시 자동 경보 `[설계목표]` |
| 죽은 소스 재시도 | 매 배치 반복 | 자동 비활성화로 차단 `[측정]` |
| 개별 소스 장애 전파 | 배치 전체 실패 위험 | 소스 단위 격리 `[측정]` |
| 경보 노이즈 | — | 1시간 쿨다운으로 중복 억제 `[설계목표]` |

## 1-6. 피드 응답 지연 — Redis 캐시

## 1-7. 추천 cold start — 협업 필터링(ALS) + 규칙 기반 폴백

**문제 (예상)**
- MVP 초기 추천은 규칙 기반(P1-4): 최근 30일 좋아요·북마크 이력의 카테고리·태그 빈도
  집계 → `80% 상위 패턴 + 20% 미접촉 카테고리`. 빈도 기반이라 **사용자 간 취향 유사성**을
  활용하지 못함.
- 반대로 협업 필터링만 쓰면 **가입 직후 이력 없는 사용자는 빈 추천**이 노출된다.

**해결**
- 협업 필터링(P6-1 / Session 7)으로 업그레이드하되, **규칙 기반을 폴백으로 유지**하는 2단 구조.
- 상호작용 가중치: `like=1.0`, `bookmark=2.0`, `view=0.3`.
- `implicit` ALS(`factors=64, iterations=20, regularization=0.1`) **일 1회 재학습**
  → user/item 팩터를 Redis에 직렬화 저장 (조회 시점 연산 부하 제거).
- 조회 시 `item_factors @ user_vec` 코사인 Top-K. 모델에 없는 신규 사용자나
  **상호작용 < 5건이면 규칙 기반 자동 폴백**.

```
pipeline/tasks/train_cf.py              — ALS 학습 + Redis 저장
backend/app/services/recommendations.py — CF Top-K + 규칙 기반 폴백
```

**결과**

| 항목 | 이전(규칙 기반) | 이후(ALS + 폴백) |
|---|---|---|
| 추천 근거 | 본인 이력 빈도만 | 유사 사용자 잠재요인(64차원) `[설계목표]` |
| Cold start 처리 | 해당 없음(항상 규칙) | < 5 상호작용 → 규칙 폴백 `[측정]` |
| 모델 갱신 | 요청 시 즉시 집계 | 일 1회 사전 학습(조회 지연 ↓) `[설계목표]` |
| 필터 버블 | 20% 강제 다양성 | 잠재요인 기반 + 폴백 다양성 `[예상]` |

## 1-8. 이종 카드 스키마 — 단일 `cards` 테이블 + nullable 컬럼

**문제 (예상)**
- NEWS(What/Why/Impact)와 TECHNIQUE(문제/아이디어/코드/주의점)는 필드 구조가 다르다.
- 타입별 테이블 분리는 피드 쿼리·인덱스·캐시·검색 색인이 전부 이원화되는 비용을 만든다.

**해결**
- **단일 `cards` 테이블 + 타입별 nullable 컬럼** 구조. `card_type` 컬럼으로 분기,
  타입별 필수 필드는 CHECK 제약(NEWS → `key_points NOT NULL`, TECHNIQUE → `problem NOT NULL`)으로 보장.
- 응답 스키마는 애플리케이션 레이어에서 `CardNewsResponse` / `CardTechResponse`로 분리.

**결과**

| 항목 | 분리 테이블(대안) | 단일 테이블(채택) |
|---|---|---|
| 피드 쿼리/커서 페이지네이션 | UNION 필요 | 단일 쿼리 `[측정]` |
| 캐시·검색 색인 | 타입별 이원화 | 공통 파이프라인 `[측정]` |
| 타입 무결성 | FK로 보장 | CHECK 제약으로 보장 `[측정]` |

# Part 2 — 실제로 일어난 문제 (운영·검증 중 발견한 버그)

> 전 항목 Playwright E2E / API 회귀 / 컨테이너 실행으로 검증 완료. `[측정]`

## 인증·프론트엔드

### 2-1. 만료 JWT 방치 → "전 기능이 안 됨" (2026-06-09)

**문제**
- JWT는 60분 만료인데 zustand `persist` 스토어가 만료된 `token`+`user`를 그대로 유지.
- 헤더는 로그인 상태로 보이지만 좋아요·북마크·추천·북마크 페이지의 모든 인증 요청이
  **401로 조용히 롤백** → 사용자에겐 "기능 전부 안 됨"으로 보임.
- 근본 원인: 만료 토큰을 감지·정리하는 로직이 프론트 어디에도 없었다.

**해결** (`frontend/stores/auth.ts`, `lib/api.ts`, `components/cards/CardActions.tsx`)
1. `isTokenExpired(token)` — JWT `exp` 디코드, `persist` 복원 시 만료면 즉시 정리.
2. `setUnauthorizedHandler()` — 토큰을 보낸 요청이 401이면 전역 logout (세션 중 만료 복구).
3. `CardActions` — 401이면 조용한 롤백 대신 `AuthModal`로 재로그인 유도.

**결과** — 만료 토큰 새로고침 시 헤더가 올바르게 "로그인" 표시, 신선 토큰으로
좋아요/북마크 정상 + 새로고침 후 유지. E2E 콘솔 에러 0.

### 2-2. 204 응답 JSON 파싱 → 좋아요/북마크 롤백 (2026-06-08)

**문제** — 좋아요/북마크 4개 엔드포인트가 204 No Content를 반환하는데
`lib/api.ts`의 `request()`가 무조건 `res.json()` 호출 → 빈 본문 파싱 예외 →
낙관적 업데이트가 롤백. **DB에는 저장되는데 UI는 원복** = "안 됨"으로 보이는 버그.

**해결** — 204 또는 `Content-Length: 0`이면 JSON 파싱 스킵. 피드 SSR이 토큰 없이
렌더돼 `is_liked`가 항상 false였던 문제도 클라이언트 토큰 재요청으로 함께 수정.

**결과** — 좋아요 ♥1 / 북마크 ■1 유지(롤백 없음), 새로고침 후 상태 유지.

### 2-3. `optional_user`가 만료 토큰에 401 → 공개 피드 전체 다운 (2026-06-09)

**문제** — 공개 피드의 선택적 인증이 만료/위조 토큰에 401을 던져,
토큰이 만료되면 **로그인 여부와 무관해야 할 피드 전체가 깨짐**.

**해결** — `dependencies.py`: 선택 인증 경로는 decode 실패 시 **게스트로 폴백**,
필수 인증 경로만 401 유지.

**결과** — 위조 토큰으로도 피드 200(게스트), 보호 엔드포인트는 여전히 401.

### 2-4. admin 가드 hydration 레이스 → 관리자 진입 불가 (2026-06-09)

**문제** — `admin/layout`이 zustand persist **재수화 전** `token=null` 상태에서
즉시 홈으로 리다이렉트 → 진짜 관리자도 admin에 못 들어감.

**해결** — `onFinishHydration` 대기 후 인증 판단.

**결과** — 관리자 진입 정상, 게스트는 여전히 리다이렉트.

## 검색

### 2-5. 인덱스 매핑 이원화 → 새 ES에서 영어/태그 검색 침묵 사망 (2026-06-10)

**문제** — 앱 시작 시 `setup_index()`가 **코드에 내장된 구버전 매핑**(`.en` 서브필드·
`tags.text`·lowercase 없음)으로 인덱스를 생성. `es/mappings/cards.json`과 이원화돼
ES를 새로 만들면 영어/태그 검색이 **에러 없이 조용히** 동작하지 않음.

**해결** — 매핑 단일 출처: `setup_index()`가 `es/mappings/cards.json`을 읽도록 통일.

**결과** — 인덱스 재생성 경로와 운영 매핑이 항상 일치. *교훈: 같은 스키마가 두 곳에 있으면 반드시 어긋난다.*

### 2-6. 검색 응답 형태 불일치 → 프론트 렌더 깨짐 (2026-06-09)

**문제** — `/v1/search`가 ES `_source` 원본을 그대로 반환 → `tags`가 문자열,
`is_liked` 누락 → React key 경고 + 태그 `#undefined` 렌더.

**해결** — 검색은 ES에서 **매칭 ID만** 얻고, 카드 본문은 DB에서 조회해 피드와 동일한
`_serialize_card` 직렬화로 반환 (검색·피드 응답 형태 통일).

**결과** — 검색 결과 카드가 피드와 동일하게 렌더, 좋아요 상태 포함.

### 2-7. 미발행 초안이 검색에 노출 (2026-06-09)

**문제** — `es/setup.py`가 `is_published=False` 초안까지 색인 → 검토 전 카드가 검색 노출.

**해결** — 색인 시 `is_published=True` 필터 추가 + 재색인. 발행 시점에만 ES 색인.

**결과** — 검색은 발행 카드만 반환.

## 파이프라인·인프라

### 2-8. Celery beat 시간대 버그 — 배치가 KST 15/21/03/09시에 실행 (2026-06-10)

**문제** — `timezone="Asia/Seoul"` 설정 시 crontab은 **KST로 해석**되는데
스케줄에 UTC 시각(15/21/3/9시)이 적혀 있어, 의도(KST 00/06/12/18시)와 달리
**KST 15/21/03/09시에 실행**되고 있었다.

**해결** — KST 시각(0/6/12/18시) 직접 기재로 수정, `train_cf`도 KST 02:00로 정정.
컨테이너 내 `remaining_estimate`로 다음 실행 시각 검증.

**결과** — KST 00/06/12/18시 + 02:00 실행 확인. *교훈: timezone 설정과 crontab 시각의 기준을 항상 함께 확인.*

### 2-9. `libgomp.so.1` 누락 → 추천 모델 학습 크래시 (2026-06-08)

**문제** — `train_cf`(implicit ALS)가 워커 컨테이너에서 `libgomp.so.1` 없음으로 크래시.
implicit는 OpenMP 네이티브 의존이 있는데 slim 이미지에 빠져 있었다.

**해결** — `pipeline/Dockerfile`에 `libgomp1` 추가, `OPENBLAS_NUM_THREADS=1`로 스레드 경고 제거.

**결과** — 재빌드 후 ALS 학습 정상. *교훈: ML 라이브러리는 파이썬 휠 밖 네이티브 의존을 컨테이너에서 검증해야 한다.*

### 2-10. B-4 큐레이션 소스의 반복 재과금 위험 (2026-06-09)

**문제** — B-4(큐레이션 CLAUDE.md 직접 수집)는 "최근 6시간" 윈도우를 쓰지 않으므로
(오래된 파일도 수집 대상), 발행 단계의 DB 중복 체크는 **AI 처리 이후**라
같은 파일에 매 배치 AI 비용이 반복 발생할 수 있었다.

**해결** — **수집 단계에서 `Card.source_url` DB 존재 여부를 선체크**
(NullPool 비동기 세션 + `run_sync`) → 이미 카드화된 파일은 네트워크·AI 비용 0으로 스킵.

**결과** — 신규 URL=수집 / 기존 URL=스킵 양방향 검증. *교훈: 비용이 드는 단계 앞에 중복 차단을 두는 원칙(§1-3)을 신규 소스에도 일관 적용.*

### 2-11. 역번역 게이트의 한계 사례 — 규칙 나열 문서의 낮은 코사인 (2026-06-09)

**문제** — 큐레이션 CLAUDE.md(card 8)의 역번역 유사도가 **0.701**로 게이트(0.85) 미달.
산문이 아닌 **규칙 나열 텍스트는 의미가 보존돼도 코사인이 낮게** 나온다 — 게이트의 구조적 한계.

**해결** — 설계해 둔 안전망이 그대로 작동: 자동 발행 차단 → 수동 검토 큐 →
Admin `approve`로 검수 후 발행(`is_published=True` + ES 색인).

**결과** — 오탐(false reject)이 사용자 노출 없이 사람 검토로 회수됨.
*교훈: 자동 게이트에는 반드시 수동 우회 경로를 함께 설계해야 한다.*

# Part 3 — 핵심 기술 및 설계 과정

## 3-1. 문서 선행 설계 → Phase 분해 → 세션 스프린트

- 구현 전에 **설계 문서 5종**(ERD·아키텍처·파이프라인 명세·API 설계·콘텐츠 소스)을 먼저 작성.
- 전체 작업을 **21개 Phase**(컨텍스트 1회에 완결되는 단위)로 분해하고 의존성 그래프로 순서 결정
  (`docs/phases.md`), 후반 작업은 **8세션 스프린트**(`docs/sessions.md`)로 묶음.
- 효과: 각 작업 단위가 "생성 파일 + 체크리스트"로 명세돼 구현·검증이 기계적으로 진행됨.

## 3-2. 파이프라인 단계 순서 — 비용이 드는 단계를 뒤로

```
수집(어댑터) → 디듀프(무료) → AI 처리(유료) → 역번역 검증 → 발행/색인
```

- **무료 필터(디듀프)를 유료 단계(Claude API) 앞에** 배치 — §1-3, §1-4, §2-10이 모두 이 원칙의 적용.
- 발행 단계에서만 ES 색인 → 초안 노출 차단(§2-7).

## 3-3. 소스 어댑터 패턴 + 단계적 그룹 도입

- `BaseAdapter.fetch() → List[RawItem]` 추상화에 RSS·GitHub Releases·GitHub Trending·
  HackerNews·IMAP·웹스크래핑 **6종 어댑터**. 신규 소스는 URL 목록 추가만으로 확장.
- 소스 그룹은 A(뉴스RSS) → B(GitHub) → C(엔지블로그) → D(뉴스레터) 순 단계 도입, 총 30+개.
- **B-2 topic 선정은 실측 기반**: GitHub API로 후보 topic 결과 수를 조회해
  `claude-md`(별≥30 → 12개, 정확) 채택, `claude`/`claude-code`(2,000+개, 노이즈) 기각.
- B-2(topic 검색)는 self-tagging 리포만 잡는 구조적 한계 → topic 없는 핵심 파일을 위해
  **B-4(큐레이션 파일 직접 수집)** 신설. 메커니즘마다 커버리지 사각을 다른 메커니즘으로 보완.

## 3-4. 신뢰성 원칙 — fail-open vs fail-closed의 일관 적용

| 지점 | 정책 | 이유 |
|---|---|---|
| 번역 품질 게이트 | **fail-closed** (미달 시 발행 차단) | 품질은 사용자 신뢰 직결 |
| 레이트리밋 (Redis 장애 시) | **fail-open** (통과) | 가용성 우선, 보조 방어선 |
| 예산 캡 (비용 조회 실패 시) | **fail-open** (캡 비활성) | 배치 중단보다 1회 과금 위험이 작음 |
| 운영 시크릿 기본값 | **fail-closed** (부팅 차단) | `change-me`로 운영 기동 금지 |

## 3-5. 보안 하드닝 (2026-06-09)

- `SecurityHeadersMiddleware` — nosniff/X-Frame-Options/Referrer-Policy/CSP/HSTS(운영).
- `RateLimitMiddleware` — Redis 고정 윈도우, IP당 300/분(기본), 429 + `Retry-After`.
  검증: 320요청 → 298×200 + 22×429 `[측정]`.
- 운영 가드 — 시크릿 기본값 부팅 차단, `/docs` 비활성, `TrustedHostMiddleware`.
- 프론트 CSP(`next.config.ts`) — 허용 출처 화이트리스트, 개발 모드만 `unsafe-eval`.

## 3-6. 인프라 구성

- **Docker Compose 6 서비스**(postgres·redis·elasticsearch·api·worker·beat) 일괄 기동.
- pipeline Dockerfile은 빌드 컨텍스트를 루트로 두고 backend 코드를 COPY,
  `PYTHONPATH=/app:/backend`로 **backend/pipeline 코드 공유** (모델·DB 세션 중복 제거).
- 클라이언트 3종: Next.js 15 SSR 웹 + React Native(Expo) iOS/Android.
- 비용 안전장치로 beat는 평시 중지(`docker compose stop beat`) — 켜면 KST 00/06/12/18시 실배치+실과금.

## 3-7. 검증 문화

- 기능 추가/수정 시 **Playwright E2E**(콘솔 에러 0 기준) + **API 회귀 16종** + 컨테이너 실행 검증.
- 실제로 §2의 버그 대부분이 "문서 기재 기능 전수 실UI 검증" 세션에서 발견됨 —
  단위 동작이 아니라 **사용자 시나리오 단위**로 검증해야 "DB엔 되는데 UI는 안 됨"류(§2-2)를 잡는다.
- 운영 수치가 없는 단계임을 인정하고 모든 수치에 `[측정]`/`[설계목표]`/`[예상]` 라벨을 유지.
