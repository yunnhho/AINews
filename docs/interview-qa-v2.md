## 1. 수집 파이프라인

### Q. 49개 소스 병렬 수집에서 한 소스가 지연되면 배치 전체는 어떻게 되나
> ✅ **이번 세션에 구현 반영**(§0.5): 아래 "정직한 정정"이 지적한 순차·무타임아웃 문제를 실제로 고쳤다.
> `collect_sources`가 **스레드풀 병렬 수집 + 소스별 타임아웃(`SOURCE_FETCH_TIMEOUT`, 기본 30s) +
> 단계 전체 데드라인(`SOURCE_FETCH_DEADLINE`, 기본 120s)** 으로 동작한다. 지연 소스는 타임아웃으로 격리되고
> 나머지는 정상 수집된다(`test_collect_sources_slow_source_times_out`로 검증). health 기록은 메인 스레드
> 단일 처리로 유지해 기존 격리 의미론을 보존했다. 아래는 개선 전 상태에 대한 설명이다.

**(개선 전) 정직한 정정**: 초기 수집은 **병렬이 아니라 순차**였다. `orchestrate.py:_run_pipeline`는
9개 그룹 콜렉터(A/B1~B4/C1/C2/D1/D2)를 **for 루프로 하나씩** 호출한다(그룹 내부가 이제 병렬).

```python
for key, collect in collectors:
    try:
        items = collect()          # ← 동기 호출, 순차
    except Exception as exc:
        items = []; failed_count += 1   # 실패는 격리, 배치는 계속
```

- **실패 격리는 된다**: 한 소스가 예외를 던져도 `try/except`로 잡아 `failed_count`만 올리고
  나머지 소스 수집은 계속 진행한다. 개별 장애가 배치 전체 실패로 번지지 않는다.
- **지연(행)에는 취약하다**: 오케스트레이션 레벨에 **소스별 타임아웃이 없다.** 한 소스가
  응답을 40초 붙들면 그 40초만큼 배치 전체가 늦어진다. 방어선은 각 어댑터의 HTTP 타임아웃뿐이고,
  그마저 어댑터마다 보장돼 있다고 말하기 어렵다.
- **개선 결과**: (1) 그룹 내부 소스 **스레드풀 병렬화** ✅, (2) 소스별 wall-clock 타임아웃 + 단계 데드라인 ✅ 구현.
  (3) 그룹 단위 서브태스크 분리/부분 재개는 아직 미구현(오케스트레이션 레벨은 여전히 그룹을 for로 순차 호출) —
  이는 Celery `group`/`chord`로 확장할 후속 과제다. 면접에서는 "격리는 처음부터, **병렬·타임아웃은 이번에 구현**,
  그룹 단위 부분 재개는 남은 과제"로 답하면 정확하다.

### Q. Celery Beat 스케줄의 KST 시간대 버그 원인과, 재발 방지를 위해 바꾼 것은
`problem-solution-result.md §2-8`의 실제 버그다.
- **원인**: `celery_app.py`에 `timezone="Asia/Seoul"`을 줬는데, crontab에는 **UTC 기준 시각(15/21/3/9시)**
  을 적어 놨다. Celery crontab은 설정된 timezone(KST)으로 해석하므로, 의도한 KST 00/06/12/18이 아니라
  **KST 15/21/03/09시**에 실행됐다. "timezone은 KST로 바꿨는데 시각은 UTC 계산값을 그대로 둔" 전형적 혼용.
- **수정**: crontab에 KST 시각을 직접 기재(`crontab(hour=0/6/12/18, minute=0)`), `train_cf`도 KST 02:00로 정정.
  컨테이너 안에서 `remaining_estimate`로 다음 실행 시각을 눈으로 검증.
- **재발 방지 원칙**: "timezone 설정과 crontab 시각의 **기준을 항상 함께** 확인한다."
  다만 이것은 규율이지 자동 가드는 아니다 — 스케줄 단위 테스트로 다음 실행 시각을 assert 하는 것이 더 견고하다(미구현).

### Q. Celery 워커가 배치 도중 죽으면 그 회차는 어떻게 복구되나
- `run_batch`는 `max_retries=2, default_retry_delay=900`(15분). 태스크가 예외로 끝나면 `self.retry`로
  최대 2회 재시도한다. `mark_batch_failed`로 `batch_logs`에 실패를 기록한다.
- **정직한 한계 3가지**:
  1. **멱등성 부족**: 파이프라인은 수집→디듀프→AI→발행을 하나의 태스크 안에서 처리한다.
     워커가 **AI 처리 도중** OOM으로 죽으면, 재시도는 **처음(수집)부터** 다시 돈다. 이미 발행된 카드는
     발행 단계의 URL 중복 체크로 걸러지지만, **AI 호출은 재과금**될 수 있다(발행 전에 죽은 카드).
  2. **acks_late 미설정 확인 필요**: 브로커에서 태스크가 소실되면 그 회차는 재시도 큐에 남지 않을 수 있다.
     현재 코드에는 `acks_late=True` / `task_reject_on_worker_lost` 설정이 명시돼 있지 않다.
  3. 복구는 "다음 6시간 배치가 직전 6시간분을 다시 긁는" 자연 복구에 상당 부분 의존한다(수집 윈도우 겹침).
- 즉 **"죽어도 다음 회차가 벌충"** 이 실질 복구 전략이며, 회차 단위 정밀 재개는 아니다.

### Q. 하루 4회 주기의 근거는? 실시간성 요구와 비용 사이의 판단
- 근거는 **콘텐츠 특성 + 비용**이다. AI/LLM 뉴스·기법은 초 단위 실시간성이 필요한 도메인이 아니고
  (속보성 주가·스포츠와 다름), 사용자가 하루 몇 번 훑는 큐레이션 피드다. 6시간 지연은 UX 손실이 작다.
- 반대편 비용: 매 배치가 Claude API 실과금이다. 실시간(웹훅/분 단위 폴링)으로 가면 호출 빈도와
  중복 처리가 급증한다. 4회는 "수집 윈도우 6시간 = 신선도"와 "일 4배치 = 비용 상한"의 절충.
- **정직한 보강점**: 4라는 숫자 자체를 A/B로 검증한 적은 없다(트래픽 부재). "왜 3도 6도 아닌 4냐"는
  질문엔 "6시간 윈도우가 소스 갱신 주기와 비용의 균형점이라 판단했고, 실데이터로 재조정할 파라미터"라고
  답하는 게 정직하다.

### Q. 소스별 파싱 형식이 제각각인데 어떻게 추상화했나
- `adapters/base.py`의 `BaseAdapter.fetch() -> list[RawItem]`로 통일. RSS·GitHub Releases·GitHub Trending·
  HackerNews·IMAP·웹스크래핑 **6종 어댑터**가 각자 형식을 파싱해 **동일한 `RawItem`**(title/url/content/
  published_at 등)으로 정규화한다. 상위 파이프라인은 소스 종류를 모른 채 `RawItem` 리스트만 다룬다.
- 소스 그룹은 A(뉴스RSS)→B(GitHub)→C(엔지블로그)→D(뉴스레터) 순으로 **단계 도입**. 신규 소스 추가는
  대개 URL 목록 추가로 끝난다.
- 설계 하이라이트: **B-2(topic 검색)** 는 self-tagging 리포만 잡는 구조적 한계가 있어, topic 없는 핵심 파일을
  위해 **B-4(큐레이션 파일 직접 수집)** 를 신설했다 — "메커니즘의 커버리지 사각을 다른 메커니즘으로 보완".

## 2. 데이터 품질 / AI 비용 제어

### Q. TF-IDF 제목 유사도 0.9 임계값은 어떻게 정했나. 서로 다른 기사를 중복으로 버린 적은 없나
- 코드 실측: `filters/dedup.py`의 `_TITLE_SIM_THRESHOLD = 0.9`. `char_wb` 분석기 + `ngram(2,3)`
  (문자 n-gram)이라 한/영 제목 모두에서 조사·띄어쓰기 차이에 강건하다. 충돌 시 **본문(content)이 더 긴 카드**를 유지.
- **0.9 산출 근거의 정직한 버전**: 통계적 튜닝(ROC로 임계 탐색)을 거친 값이 **아니다.** char n-gram 코사인에서
  0.9는 "제목이 거의 동일한 재게시"만 잡고 다른 기사는 살리는 **보수적(재현율 희생, 정밀도 우선)** 지점으로
  택한 휴리스틱이다. 중복을 놓치더라도(미탐) 서로 다른 기사를 버리는 것(오탐)이 사용자에게 더 나쁘다는 판단.
- **오탐 위험은 실재한다**: `"GPT-5 출시"` vs `"GPT-5 출시 임박설"`처럼 접두가 겹치는 제목은 char n-gram
  유사도가 높게 나와 오탐 가능. 현재는 0.9 상향 + "긴 본문 보존"으로 완화했을 뿐, **오탐률을 측정한 적은 없다.**
- **문서 모순 주의**: `pipeline.md`=0.9, `problem-solution-result.md §1-3`=0.85로 문서가 엇갈린다. **코드가 정답(0.9).**

### Q. URL 완전 일치 디듀프가 놓치는 케이스(리다이렉트, UTM)는 어떻게 처리했나
> ✅ **이번 세션에 URL 정규화 구현**(§0.5): `dedup.py:_normalize_url`이 해시 전 **스킴/호스트 소문자화 +
> 추적 파라미터(`utm_*`·`fbclid`·`gclid` 등) 제거 + 프래그먼트 제거 + 말미 슬래시 정리**를 수행한다.
> `?utm_source=...` 변형과 `#fragment` 변형은 이제 동일 URL로 중복 제거된다
> (`test_dedup_normalizes_tracking_params`로 검증). 실질 쿼리(`?id=2`)는 보존한다.
- **정직하게 남은 한계**: (1) **리다이렉트 최종 URL 해석은 여전히 미구현**(HTTP HEAD로 resolve하려면 네트워크
  비용·타임아웃 관리 필요) — 리다이렉트 경유로 URL이 다르면 못 잡는다. (2) URL 정규화는 **배치 내부** 해시에
  적용되고, DB 대조(과거 카드)는 저장된 원본 `source_url` 기준이라 **배치를 넘는 UTM 변형**은 여전히 통과 가능.
- 원본 URL 자체는 카드에 **정규화하지 않고 보존**한다(CLAUDE.md "원본 URL 노출" 규칙 준수) — 정규화 값은 오직
  중복 판정 해시에만 쓴다.

### Q. 역번역 검증은 호출 비용이 2배가 되는데, 그 비용을 정당화한 근거는
- 실제로 2배는 아니다. `backtranslate.py`는 **KO 요약문만**(원문 전체가 아니라) 다시 EN으로 역번역하고,
  `max_tokens=512`로 상한을 건다. 요약은 원문보다 짧으므로 추가 비용은 "요약 1회 번역분"에 가깝다.
- 정당화: 번역 필드(요약·문제·아이디어)가 곧 서비스 품질이고, 환각 번역이 발행되면 신뢰가 직접 깨진다.
  **사람 전수 검수는 배치 자동화의 의미를 없앤다.** 역번역 게이트는 "저품질만 사람에게" 보내는 필터라
  검수 인건비 대비 API 추가비가 싸다는 판단.
- 비용 절감 장치: 요약문만 검증 + `[:500]`자 절단 임베딩 + `sentence-transformers`는 **로컬(무료)**.
  유료는 역번역 Claude 호출뿐.

### Q. 코사인 0.85 게이트의 정확도를 실제로 측정해봤나 (오탐/미탐)
- **정직하게: 통계적으로 측정하지 않았다.** 가진 것은 소표본 관찰뿐이다.
  - 첫 실배치 5건: 2건 통과(0.887 / 0.928), 3건 검토 큐(0.70~0.79). `[측정]`
  - 알려진 **구조적 오탐 사례**(`§2-11`): 큐레이션 CLAUDE.md처럼 **산문이 아닌 규칙 나열 텍스트**는 의미가
    보존돼도 코사인이 낮게 나온다(0.701 → 게이트 미달). 이건 임베딩 모델이 문장 의미쌍에 최적화된 데서 오는
    체계적 편향이다.
- 즉 오탐/미탐 **비율**은 모른다. 대신 **오탐을 회수하는 안전망**을 설계로 보장했다: 미통과 → 자동 발행 차단 →
  수동 검토 큐 → Admin `approve`. "게이트가 완벽하지 않다는 전제 위에 사람 우회로를 깔았다"가 핵심.
- 운영 가시성: 번역 통과율 KPI + **통과율 < 70%면 경보**(`alerting.py`, 24h 쿨다운)로 게이트가 과하게
  조여지는지 감시.

### Q. 3,000자 절단으로 핵심 내용이 잘리는 케이스는 어떻게 다뤘나
- 절단은 **비용 상한**을 위한 의도적 선택이다(`content[:3000]` → 입력 토큰 고정). 뉴스/블로그의 핵심은
  대개 상단에 있다는 역피라미드 가정.
- **정직한 한계**: 긴 기술 문서에서 결론이 뒤에 있으면 잘린다. 이를 **감지·보정하는 로직은 없다.**
  요약 품질이 낮아지면 역번역 게이트가 일부 걸러줄 수는 있으나(잘린 요약이 원문과 유사도 하락),
  "절단 때문에 놓친 핵심"을 직접 잡는 장치는 아니다. 개선안: 소스 타입별 절단 길이 차등, 또는 청킹 요약.

### Q. 월 예산 하드 캡에 도달하면 서비스는 어떤 상태가 되나
- `orchestrate.py:_process_items`: 배치 시작 시 이달 `batch_logs` 누적 비용을 조회하고, AI 처리 루프가
  `이번 배치 누적 ≥ (예산 − 이달 누적)`에 도달하면 **이후 항목 AI 처리를 중단**한다.
- 중요한 성질:
  - **수집·디듀프(무료)는 계속** 돈다. 멈추는 건 유료(Claude) 단계뿐.
  - **읽기 서비스(피드/검색/추천)는 영향 없다** — 이미 발행된 카드는 그대로 서빙. 캡은 "신규 카드 생성"만 멈춘다.
    사용자 체감은 "새 카드가 안 올라온다"이지 "서비스 다운"이 아니다.
  - 비용 조회 실패 시 **fail-open**(캡 비활성) — "배치 1회 초과 과금" < "가용성 손실"이라는 판단(`§3-4`).
- 현재 캡 $20(env `MONTHLY_BUDGET_USD`, config 기본값은 100). 페이크 수집기 테스트: 예산 0 → AI 호출 0회 검증. `[측정]`

### Q. 카드 1건 $0.0037은 어떻게 실측했나. 모델이 바뀌면 이 구조는 유지되나
- 산출: `claude-haiku-4-5` 요금(input $0.80/1M, output $4.00/1M, `orchestrate.py`에 상수로 박음)에
  Claude 응답의 `usage.input_tokens/output_tokens`를 곱해 카드별 비용을 계산하고 `batch_logs`에 누적.
  $0.0037은 그중 **B-4 카드 1건의 실측 토큰**으로 계산한 값. 5건 테스트 배치 총액 $0.0122도 같은 방식. `[측정]`
- **정직한 한계**: 이건 **단일 표본 1건**이다. 분산·평균·상한이 아니다. 긴 원문·코드 스니펫이 많은 카드는
  더 비싸다. "카드당 평균 비용"으로 일반화하려면 표본이 필요하다.
- **모델 변경 내성**: 토큰 회계(usage × 단가) 구조 자체는 모델 불변으로 유지된다. 다만 **단가 상수(`_COST_IN/_COST_OUT`)
  가 코드에 하드코딩**돼 있어, 모델을 바꾸면 이 상수를 함께 고쳐야 정확하다. 절대 금액($0.0037)은 당연히 바뀐다.
  개선안: 모델→단가 매핑 테이블로 분리.

### Q. 카드 타입별 프롬프트 분리 기준은? 프롬프트 변경 시 회귀 검증은
- 기준: **출력 스키마가 다르다.** NEWS=What/Why/Impact + key_points, TECHNIQUE=문제/아이디어/코드/주의점/선행지식.
  `ai/prompts.py`에서 타입별 시스템·유저 프롬프트를 분리해, 불필요한 지시 토큰을 줄이고 파싱 스키마를 고정.
  타입 판정은 코드블록/GitHub 링크/"how to"→TECHNIQUE, 그 외→NEWS(불확실 시 안전 기본값 NEWS).

  현재는 `parse_json_response`가 불완전 JSON을 복구하는 방어 파싱 + 발행 스키마 검증에 의존한다.
  "프롬프트를 바꾸면 무엇이 깨지는지"를 잡는 골든 데이터셋 회귀는 향후 과제.

## 3. 검색

### Q. Meilisearch에서 Elasticsearch로 옮긴 실제 이유는. 처음 선택이 틀린 이유는
- 전환 이유: 한국어 검색 품질 + 복합 필터. Meilisearch는 초기 MVP엔 가벼워 좋았지만, 한국어 토크나이징과
  카테고리×타입×난이도×태그 **복합 term 필터** 조합에서 한계가 있었다. 카드의 요약·문제·아이디어가 전부
  **한국어 번역문**이라 검색 품질이 곧 서비스 품질이었다.
- **"첫 선택이 틀렸다"에 대한 정직한 프레이밍**: 완전한 실패는 아니라 **MVP 단계 적정 선택 → 요구 확장으로 재평가**다.
  검색이 핵심 기능으로 승격되면서 필터 조합·언어 분석 요구가 커졌고, 그 시점에 엔진을 바꿨다. "처음부터 ES를 썼어야"
  라기보다 "검색 비중이 커질 걸 초기에 과소평가했다"가 더 정확한 반성이다.

### Q. 한국어 형태소 분석기(nori) 대신 cjk 바이그램을 쓴 이유와 그로 인한 검색 품질 차이는
**이 질문이 코드의 진실을 정확히 찌른다. 문서(§1-1)의 "nori 형태소" 서술은 실제와 다르다.**
- 실제 구현(`es/mappings/cards.json`): analyzer 이름은 `korean`이지만 내용은
  `tokenizer: standard` + 필터 `[cjk_width, lowercase, cjk_bigram]` — 즉 **형태소가 아니라 CJK 바이그램**.
  클라이언트도 `opensearchpy.AsyncOpenSearch`(ES 클라이언트 아님).
- **왜 nori가 아니라 cjk인가(정직한 이유)**: 배포 대상이 **Bonsai 무료 티어(OpenSearch)** 다. 무료 관리형에서는
  `analysis-nori` 같은 플러그인을 설치할 수 없다. nori를 쓰려면 플러그인 포함 ES 이미지를 직접 빌드·자체 호스팅해야
  하는데(레포에 `es/Dockerfile`이 그 잔재로 남아 있으나 **런타임에서 미사용**), 비용·운영 부담이 커서
  **플러그인 불필요한 cjk_bigram**으로 갔다.
- **품질 차이(정직한 트레이드오프)**:
  - cjk_bigram은 `"임베딩"`을 `임베/베딩` 2-gram으로 쪼갠다. 조사 변형(`임베딩을`/`임베딩이`)에도 겹치는
    bigram이 생겨 **웬만한 재현율은 확보**된다. 플러그인 0, 사전 관리 0이라는 게 최대 장점.
  - 단점: 형태소 경계를 모르므로 **정밀도가 낮다.** `"모델"`이 `"모델링"`·`"모델하우스"`에도 bigram이 겹쳐
    걸릴 수 있고, 어근 정규화(`먹었다`→`먹다`)가 안 된다. nori라면 품사 필터로 조사·어미를 떼어 더 정확했을 것.
  - 요약: **"플러그인·사전 없이 그럭저럭 되는 재현율"을 얻는 대신 "형태소 수준 정밀도"를 포기**했다.
- 면접 답변 요지: "문서에는 nori라고 적혀 있지만 **실제 운영 제약(Bonsai 무료 = 플러그인 불가)** 때문에
  cjk 바이그램을 택했고, 그게 정밀도-운영비용 트레이드오프의 결과다"라고 **정정하며** 답하는 것이 정확하다.

### Q. 색인 클라이언트 불일치가 프로덕션에서 무음 실패했다는데, 어떻게 발견했고 감지 장치를 뭘 추가했나
- 실제 버그(`§2-5`): 앱 시작 `setup_index()`가 **코드에 내장된 구버전 매핑**(`.en` 서브필드·`tags.text`·lowercase 없음)으로
  인덱스를 만들었다. `es/mappings/cards.json`과 **이원화**돼, ES를 새로 만들면 **영어/태그 검색이 에러 없이 조용히** 안 됐다.
- **발견 경로(정직하게)**: 자동 감지가 잡은 게 아니다. **"문서 기재 기능 전수 실UI 검증" 세션**에서 사람이
  "영어/태그 검색 결과가 비는" 걸 눈으로 보고 발견했다. 예외도 로그도 안 나는 전형적 무음 실패.
- **추가한 "장치"**: 근본 수정은 **매핑 단일 출처화** — `setup_index()`가 `es/mappings/cards.json`을 읽도록 통일해
  이원화 자체를 없앴다(`search.py`의 `_MAPPING = json.loads(_MAPPINGS_FILE.read_text())`).
- **정직한 미흡점**: 이건 "재발 원인 제거"이지 "감지기 추가"가 아니다. 매핑 드리프트/색인 정합성을 **런타임에 감지하는
  헬스체크는 아직 없다**(예: 부팅 시 실제 인덱스 매핑 vs 파일 매핑 diff, 색인 문서 수 vs DB 발행 카드 수 대조).
  "무음 실패를 겪고 배운 교훈"으로 이 자동 대조를 후속 과제로 두는 게 솔직한 답이다.

### Q. 무중단 재색인(alias 스위칭)은 고려했나
- **정직하게: 미구현.** `setup_index()`는 고정 인덱스명 `cards`에 대해 "없으면 생성"만 한다. alias(`cards_v1`→`cards_v2`)
  스위칭, 재색인 완료 후 원자적 alias 전환은 없다. 매핑을 바꾸려면 인덱스를 지우고 다시 만들어야 하고, 그 사이 검색은 빈다.
- 완화: 검색은 **ES에서 매칭 ID만** 얻고 카드 본문은 DB에서 조회(`_hydrate_cards_from_db`)하므로, 재색인 중이라도
  **DB 데이터는 안전**하다(검색만 잠시 빔). 하지만 "무중단 재색인"이라 부를 수는 없다.
- 개선안: 버전드 인덱스 + alias 스위칭 + 재색인 백필 잡. MVP·저트래픽이라 우선순위를 낮췄다는 게 정직한 배경.

### Q. 멀티필드 검색에서 필드별 가중치는 어떻게 줬나
- `search.py`의 `multi_match`, `type: best_fields`, `operator: or`:
  - `title^3`, `title.en^3` (제목 최우선)
  - `summary^2`, `summary.en^2`, `tags.text^2`
  - `problem`, `problem.en`, `idea`, `idea.en` (가중치 1)
- 한국어(cjk analyzer) 본필드 + 영어(`english` analyzer) `.en` 서브필드 + 태그(`standard`) 서브필드를 **동시 검색**해
  한/영/키워드를 한 쿼리로 커버. 정렬은 `["_score", {published_at: desc}]` — 관련도 우선, 동점이면 최신순.
- **정직한 한계**: 가중치(3/2/1)는 "제목>요약·태그>본문"이라는 상식적 서열의 휴리스틱이고, 클릭 로그 기반
  튜닝(learning-to-rank)은 없다. `best_fields`라 여러 필드 매칭을 합산하지 않고 최고 점수 필드만 반영한다는 점도 인지 필요.

### Q. DB와 색인의 정합성은 무엇으로 보장하나
- 쓰기 경로: 카드 **발행 시점**에만 ES 색인(`index_card`), 삭제 시 `delete_card`. 초안(`is_published=False`)은
  색인 안 함(`§2-7` 수정). 조회 시에도 `_hydrate_cards_from_db`가 `is_published=True`만 재확인 → **미발행 노출 이중 차단**.
- **정직한 한계(중요)**: DB와 ES는 **분산 트랜잭션이 아니다.** DB 커밋은 됐는데 ES 색인 호출이 실패하면
  (네트워크·ES 다운) **그 카드는 검색에 안 뜨는 드리프트**가 생긴다. 현재 이를 **주기적으로 대조·복구하는
  reconcile 잡이 없다.** `es/setup.py`의 전체 재색인을 **수동**으로 돌려야 맞춰진다.
- 그나마의 안전판: 검색이 "ID만 ES, 본문은 DB"라 **DB가 단일 진실원**이다. ES는 "발행 카드의 검색 인덱스"일 뿐,
  손상돼도 재색인으로 완전 복원 가능(원본은 항상 DB). 정합성의 **최종 보증은 "DB 기준 재색인 가능성"**이지
  "실시간 일치 보장"이 아니다.

### Q. ALS 모델 재학습 주기와 오프라인 평가 지표는
- 재학습: **일 1회 KST 02:00**(`celery_app.py` beat + `train_cf.py`). `implicit` ALS
  `factors=64, iterations=20, regularization=0.1`. user/item 팩터를 pickle로 Redis에 **2일 TTL** 저장.
- 상호작용 가중치: like=1.0, bookmark=2.0(view는 문서엔 0.3이나 `train_cf.py` 학습엔 like/bookmark만 반영 — **문서-코드 소차이**).
- **오프라인 평가 지표: 정직하게 없다.** `train_cf.py`는 fit 후 곧바로 Redis에 저장할 뿐,
  train/test split도, precision@k·recall@k·MAP·NDCG 계산도 **하지 않는다.** "재학습은 하지만 평가는 안 한다"가 현실.
  이건 명백한 갭이고, 프로덕션 전 반드시 채워야 할 부분이다.

### Q. 상호작용 5건 기준의 근거는
- `recommendations.py:_MIN_INTERACTIONS_FOR_CF = 5`. 상호작용 < 5건이거나 모델에 없는 신규 사용자면 규칙 기반 폴백.
- **정직한 근거**: 통계적 최적화값이 아니라 **cold-start 휴리스틱**이다. "잠재요인이 의미를 가지려면 최소 몇 건은 필요하다"는
  상식선에서 5로 잡았다. 3이나 10이 더 나을 근거도 딱히 없다 — 실데이터로 튜닝할 파라미터.

### Q. 추천 품질을 어떻게 확인했나. 규칙 기반 폴백보다 낫다는 근거가 있나
- **가장 정직해야 할 질문.** 결론: **"ALS가 규칙 기반보다 낫다"는 것을 입증한 근거가 현재 없다.**
  - 오프라인 지표 없음(위), A/B 테스트 없음, 프로덕션 트래픽 없음.
  - 트래픽이 없으니 상호작용 데이터도 거의 없고, `_MIN_INTERACTIONS_FOR_CF=5` 문턱을 넘는 사용자가 사실상 없어
    **실전에선 거의 항상 규칙 기반으로 폴백**된다. 즉 지금 ALS는 "구현·배선은 됐지만 실효는 검증 전"인 상태.
- 검증된 것은 **경로(plumbing)뿐**이다: 모델 학습→Redis 저장→조회 시 Top-K 산출→cold-start 폴백 분기가 코드로 동작.
- 면접 답변 요지: "추천 **품질**은 아직 검증하지 못했다. 데이터가 쌓이면 (1) leave-one-out 오프라인 평가,
  (2) 규칙 기반 대비 CTR A/B로 우열을 실제로 재겠다"가 정직하고 성숙한 답이다. 없는 성과를 지어내지 않는 게 핵심.

### Q. 사용자·아이템이 계속 늘어날 때 학습 비용은 어떻게 감당하나
- 현재 구조의 한계를 솔직히: `train_cf.py`는 **매일 전체 상호작용을 메모리에 올려 full-refit** 하고, user/item
  팩터를 `.tolist()`로 통째 직렬화해 Redis에 넣는다. 사용자·아이템이 커지면
  (1) 희소행렬·팩터 메모리, (2) 매일 full-refit 시간, (3) pickle 크기·Redis 메모리가 모두 **선형~그 이상으로 증가**.
- MVP 규모(수십~수백 사용자)에선 수초 학습으로 충분하지만, **확장 전략은 아직 없다.** 개선안:
  incremental/partial fit, 팩터를 Redis 통짜 대신 사용자별 키로 분산, 학습 빈도·샘플링 조정, 전용 학습 워커 분리.
- "지금은 문제없지만 스케일 대비는 미설계"가 정직한 답.

## 5. 캐싱 / API

### Q. 필터 조합별 캐시 키는 조합 폭발이 생기지 않나. 무효화 전략은
- 캐시 키(`cards.py:_cache_key`): `feed:{category}:{card_type}:{tags}:{difficulty}:{cursor}:{limit}`.
  태그는 `sorted()` 후 join하므로 순서 무관하게 정규화된다.
- **조합 폭발은 실재한다(정직하게)**: category × card_type × **태그 부분집합** × difficulty × **cursor** × limit.
  특히 tags 부분집합과 cursor(연속값)가 곱해져 키가 많아질 수 있다.
> ✅ **이번 세션에 무효화 구현**(§0.5): **버전 네임스페이스** 방식을 넣었다. 모든 캐시 키에 `feed:ver` 값을
> 접두어로 붙이고(`f"{ver}:feed:..."`), 카드 발행 시 `invalidate_feed_cache()`가 `feed:ver`를 **INCR**한다
> (`publisher.py`가 발행 성공 직후 호출). → 폭발한 키를 SCAN/DEL로 훑지 않고 **O(1)로 전체 무효화**,
> 기존 키는 참조 불가 상태로 TTL 만료돼 자연 소멸. 문서(pipeline ⑦)의 "캐시 무효화" 서술이 이제 코드와 일치한다.
- **무효화 계층**: (1) 발행 시 버전 INCR로 즉시 무효화, (2) 그와 별개로 각 키 5분 TTL이 안전망. best-effort라
  Redis INCR가 실패해도 TTL이 방어한다.
- 남은 리스크: 폭발한 키가 TTL(5분) 동안 메모리를 점유하는 것 자체는 여전하다(무효화는 "참조 차단"이지 즉시 삭제는 아님).
  트래픽이 커지면 태그 조합 제한 등 카디널리티 관리가 추가로 필요.

### Q. TTL 5분 동안의 stale 데이터는 허용 가능한 요구사항이었나
- 그렇다, **도메인 특성상 허용**된다. 배치가 KST 6시간 주기라 콘텐츠 자체가 6시간 단위로만 바뀐다.
  피드에서 5분 지연은 신선도 손실이 사실상 0이다(`§1-6`).
- **정직한 예외**: 캐시는 **비개인화 카드 본문만** 담고(`_serialize_card(c, set(), set())`), 좋아요/북마크 상태(`is_liked`
  /`is_bookmarked`)는 **캐시 히트 후에도 매번 사용자별로 다시 계산**(`_get_user_interaction_ids`)한다. 즉 "내가 방금 누른
  좋아요"는 stale하지 않다 — 카드 목록만 5분 stale하고 **개인 상호작용 상태는 실시간**이다. 이 분리가 TTL 5분을
  안전하게 만든 이유다.

### Q. 커서 페이지네이션의 커서를 무엇으로 구성했나. 정렬 값이 동일할 때 안정성은
> ✅ **이번 세션에 복합 커서로 수정**(§0.5): 아래 지적한 동점 누락 버그를 실제로 고쳤다.
> 커서 = **`published_at.isoformat()|id`**, 쿼리는 `WHERE (published_at, id) < (:ts, :id)`(튜플 비교) +
> `ORDER BY published_at DESC, id DESC`. 구(舊) 형식 커서(타임스탬프 단독)도 `_decode_cursor`가 하위호환
> 처리한다(`cards.py`). id 타이브레이커로 **동점에서도 무손실·안정**.

- 커서 = **복합 키 `(published_at, id)`**. `_encode_cursor`가 `ISO|id`로 직렬화, `_decode_cursor`가 파싱.
- **(개선 전) 약점**: 커서가 **단일 타임스탬프**였을 땐 정렬 값 동점에서 불안정했다.
  - 같은 `published_at` 카드가 페이지 경계에 걸치면 `< cursor_dt`(strict)가 **동일 타임스탬프 행 전부를
    건너뛰어** 누락, `<=`면 중복. 배치가 여러 카드를 사실상 동시 INSERT하면 초 단위 겹침이 실제로 발생 → 위험이었다.
- **수정 방식**: `tuple_(Card.published_at, Card.id) < (cursor_dt, cursor_id)` 행-값 비교로 id가 타이브레이커가 되어
  동점 경계에서도 정확히 이어진다. 검색 경로(offset 기반)는 영향 없음.

### Q. 커서 방식에서 사용자가 특정 페이지로 점프해야 한다면 어떻게 대응하나
- 커서 페이지네이션은 **임의 페이지 점프(예: 5페이지로)를 원리적으로 지원하지 않는다.** 커서는 "직전 페이지 마지막
  값"에서만 이어갈 수 있어 순차 앞/뒤 이동에 최적이고, "N페이지"에는 부적합하다.
- 피드는 **무한 스크롤**이 UX라 이 제약이 자연스럽다(점프 요구 자체가 거의 없음). 만약 점프가 필요하면:
  (1) 그 화면만 **offset/limit 페이지네이션**을 별도 제공(딥 offset 성능 저하 감수), (2) 또는 검색 경로처럼
  `from/size` 방식 사용. 실제로 `search_cards`는 이미 **offset/limit(`from_`/`size`)** 을 쓰므로, 점프가 필요한
  맥락(검색 결과 페이지)에는 offset 방식이 이미 존재한다. **"피드=커서(스크롤), 검색=offset(점프 가능)"** 로
  용도에 맞게 나눴다고 답할 수 있다.

## 6. 장애 격리 / 운영

### Q. 3회 연속 실패로 소스를 비활성화할 때, 일시적 장애를 영구 차단할 위험은
- **정확히 정정할 지점**: 코드상 **3회 연속 실패는 "비활성화"가 아니라 "경보"** 다.
  - `alerting.py:check_and_alert`: `consecutive_failures >= 3`인 소스에 **Slack/이메일 경보**(1h 쿨다운).
  - **자동 비활성화(`enabled=False`)는 404 등 영구 실패에만**(`source_health.py:record_failure(disable_on_404=True)`).
- 따라서 "일시 장애를 3회로 영구 차단"하는 일은 설계상 일어나지 않는다 — 일시 장애는 **경보만** 울리고 재시도는 계속된다.
  다음 배치에서 성공하면 `record_success`가 `consecutive_failures=0`으로 리셋.
- **정직한 잔여 위험**: 404 자동 비활성화는 "일시적 404"(오배포·CDN 순단)도 영구 차단할 수 있다. 이 오판은
  아래 질문의 "자동 복구 부재"와 맞물려 실제 리스크가 된다.

### Q. 404 자동 비활성화된 소스는 다시 살아나면 어떻게 복구되나
- **정직하게: 자동 복구되지 않는다.** 두 가지가 겹친다.
  1. `record_success`는 기존 행에 대해 `consecutive_failures=0`만 리셋하고 **`enabled=True`로 되돌리지 않는다.**
  2. 애초에 비활성 소스는 수집 루프가 `get_disabled_sources()`로 건너뛰므로 **성공을 기록할 기회조차 없다**(닭-달걀).
- 복구는 **Admin 수동 재활성화**로만 된다: `admin.py:toggle_source(source_id, enabled=True)` (Admin 대시보드에서 토글).
- **정직한 개선안**: (1) 비활성 소스도 저빈도(예: 하루 1회) **프로브 재시도**해 살아나면 자동 재활성화,
  (2) 404를 "N회 연속 404"로 완화해 순간 404의 오비활성화 방지. 현재는 "안전하게 끄고 사람이 켠다"는 보수적 정책.

### Q. Slack/이메일 경보의 노이즈는 어떻게 관리했나
- alert_type별 **Redis 쿨다운**(`alerting.py:_COOLDOWN_TTLS`): source_failure 1h, zero_collection 1h,
  low_translation_rate 24h. 같은 (type, source) 키가 쿨다운 중이면 재발송을 억제한다.
- 소스별로 키가 분리(`alert_cooldown:{type}:{source}`)돼, 한 소스의 쿨다운이 다른 소스 경보를 막지 않는다.
- 발송 이력은 `AlertLog`에 남겨 Admin에서 조회. `send_test_alert`는 `force=True`로 쿨다운 무시(수동 점검용).
- **정직한 한계**: 쿨다운은 "고정 시간 억제"라, 장애가 1시간+1분마다 재발하면 매시간 1건씩은 계속 울린다.
  플래핑(짧게 실패/성공 반복) 소스에 대한 지수 백오프나 그룹 집계 경보는 없다.

### Q. Railway·Vercel 분리 배포에서 환경 차이로 겪은 문제는
- 실제로 부딪힌 **환경 차이 버그들**(`§2-8~2-13`)이 대부분 여기에 해당한다:
  - **`libgomp.so.1` 누락**(`§2-9`): `implicit`(ALS)가 OpenMP 네이티브 의존인데 slim 이미지에 빠져 워커 컨테이너에서
    크래시. `libgomp1` 추가 + `OPENBLAS_NUM_THREADS=1`로 해결. → "파이썬 휠 밖 네이티브 의존은 컨테이너에서 검증".
  - **alembic이 컨테이너에서 실패**(`§2-12`): `env.py`가 localhost DSN 고정 → `DATABASE_URL` env 우선 읽도록 수정.
  - **`next build` 실패**(`§2-12`): 미사용 import가 ESLint 에러로 프로덕션 빌드만 차단(dev 서버는 통과).
  - **cookie 관련 배포 설정**: `COOKIE_SECURE`/`COOKIE_DOMAIN=.aipulse.kr`/`SameSite`와 `TRUSTED_PROXY_COUNT`를
    배포 토폴로지(LB 홉 수)에 맞춰야 인증/레이트리밋이 정상 동작(보안 감사 배포 체크리스트).
- 공통 교훈: **"dev 서버만 돌리지 말고 프로덕션 빌드·컨테이너 경로를 주기적으로 통과"**(`§2-12`). 환경 차이 버그는

## 7. 보안

### Q. localStorage 대신 HttpOnly 쿠키로 바꾼 이유와, 그때 새로 생긴 문제(CSRF)를 어떻게 막았나
- **바꾼 이유**(보안 감사 #1·#2): 기존엔 JWT를 **localStorage + zustand persist**에 저장(`§2-1~2-4`는 그 시절 버그들)했고,
  OAuth 콜백이 **토큰을 URL 쿼리스트링**으로 넘겼다. → XSS 한 번에 토큰 전량 탈취 + URL이 히스토리·Referer·로그에 잔류.
- **전환**: access/refresh를 **HttpOnly + Secure + SameSite 쿠키**로만 다룬다(`cookies.py`). JS에서 토큰 접근 불가 →
  XSS로도 못 훔친다. 프론트 스토어는 **비민감 프로필만** 보관, 모든 fetch는 `credentials:'include'`.
- **새로 생긴 CSRF를 막은 2중 방어**:
  1. **SameSite=lax**: 운영에서 `aipulse.kr`↔`api.aipulse.kr`이 동일 사이트라, 교차 사이트 POST엔 쿠키가 안 실린다(1차).
  2. **더블 서브밋 토큰**(`middleware.py:CsrfMiddleware`): 비-HttpOnly `csrf_token` 쿠키를 프론트가 읽어
     `X-CSRF-Token` 헤더로 되돌려보내고, 서버가 `secrets.compare_digest`로 쿠키==헤더 일치를 검증(2차).
     공격자 도메인은 피해자의 csrf 쿠키를 읽을 수 없어 헤더를 못 만든다.
- 안전 메서드(GET/HEAD/OPTIONS)와 **Bearer 인증(모바일)** 은 CSRF 면제.

### Q. 더블 서브밋 쿠키의 한계는? SameSite 속성만으로 부족한 이유는
- **SameSite만으로 부족한 이유**: (1) 구형 브라우저는 SameSite 미지원/기본값 상이, (2) `lax`는 **top-level GET 네비게이션**엔
  쿠키를 실어주므로 GET 기반 상태변경이 있으면 취약(그래서 unsafe 메서드로만 상태변경하도록 설계), (3) **서브도메인
  구성**에 따라 "same-site" 판정이 느슨해질 수 있다. 그래서 토큰 기반 2차 방어를 덧댄다.
- **더블 서브밋의 정직한 한계**:
  - **서브도메인 쿠키 주입/오염**: 같은 상위 도메인의 다른 서브도메인이 XSS로 뚫리면 부모 도메인 csrf 쿠키를
    덮어써 우회 시도가 가능하다(쿠키는 origin이 아니라 domain 스코프).
  - **비-HttpOnly가 전제**: csrf 토큰은 JS가 읽어야 하므로 HttpOnly가 아니다. 즉 **XSS 앞에서는 CSRF 토큰 자체가
    탈취**된다(더블 서브밋은 CSRF 방어지 XSS 방어가 아니다). XSS 방어는 access 토큰을 HttpOnly로 숨긴 것 + 프론트 CSP가 담당.
  - 상태 저장형(서버 세션에 토큰 보관, Synchronizer Token) 대비 **서버가 토큰을 기억하지 않아** 검증이 약하다 —
    "쿠키와 헤더가 같기만 하면 통과"라 쿠키를 심을 수 있는 공격면이 있으면 뚫린다.
- 종합: SameSite(1차) + 더블 서브밋(2차) + HttpOnly access(XSS 완화)를 **겹쳐** 단일 실패점을 없앴다는 게 설계 요지.

### Q. React Native 앱은 쿠키 기반 인증을 어떻게 처리했나 (웹과 다른 방식인가)
- **다르다.** 모바일은 쿠키를 쓰지 않고 **Bearer 토큰(JWT)** 방식이다.
  - 로그인: `_complete_login`이 platform=mobile이면 **일회용 코드**를 딥링크(`aipulse://auth/callback?code=...`)로 전달.
  - 교환: 앱이 `POST /v1/auth/exchange {code}` → `{access_token, refresh_token}` **JSON**으로 수령(코드 5분 TTL, `getdel` 원자적 소비).
  - 이후: `Authorization: Bearer <access>` 헤더로 API 호출. 토큰은 앱 보안 저장소(예: SecureStore)에 보관.
- 서버는 access를 **Bearer 헤더 또는 쿠키 양쪽**에서 추출(`dependencies.py`)해 웹·모바일을 한 API로 지원.
  Bearer 요청은 **CSRF 면제**(쿠키 인증이 아니라서 CSRF 대상이 아님).
- **정직한 잔여 과제**: 모바일 OAuth에 **PKCE 미적용**이라 `aipulse://` 커스텀 스킴 가로채기 위험이 남아 있다
  (보안 감사 후속 과제). 일회용 코드 5분 TTL + 원자적 소비로 창을 좁혔을 뿐이다.

### Q. X-Forwarded-For 신뢰 범위(trusted proxy hop)는 어떻게 설정했나
- `middleware.py:RateLimitMiddleware._client_ip` + `settings.TRUSTED_PROXY_COUNT`(기본 **0**).
  - hops ≤ 0: **XFF 무시**, `request.client.host`(직접 연결 IP)만 사용.
  - hops = N: XFF는 "각 프록시가 자기가 본 IP를 끝에 덧붙인다"는 성질을 이용해, **끝에서 N번째** 값을 실제 클라이언트로 채택
    (`idx = len(parts) - hops`).
- **이유**: XFF는 클라이언트가 위조 가능하다. 무조건 첫 IP를 믿으면 매 요청 가짜 IP로 버킷을 분산해 레이트리밋을
  무력화한다(보안 감사 #5). "신뢰 프록시 홉 수만큼만 뒤에서 잘라 쓴다"가 핵심. 직접 노출 배포면 0으로 둬 XFF를 전면 불신.
- 검증(`security-test-scenarios F`): `TRUSTED_PROXY_COUNT=0`에서 `X-Forwarded-For`를 매번 위조해도 동일 버킷 →
  11번째부터 429(우회 불가) `[측정]`. **주의**: 이 값은 배포 토폴로지(LB 홉 수)에 맞춰 **정확히** 설정해야 한다.
  너무 크면 프록시가 붙인 값을 클라이언트로 오인, 너무 작으면 위조에 노출.

### Q. 발견한 취약점 8건 중 수정하지 않은 1건은 무엇이고, 왜 두었나
- 미수정 1건 = **#8 HS256 대칭키 단일 시크릿 + 키 회전(kid) 메커니즘 없음** — 상태 `📝 수용(Accepted Risk)`.
- **문제**: 시크릿이 노출되면 전체 토큰 위조가 가능하고, 무중단 키 회전을 못 한다.
- **왜 두었나(정직한 근거)**: (1) 시크릿은 환경변수로만 관리되고 커밋된 값이 없다, (2) **운영 부팅 시 기본값(`change-me`)을
  차단**해 약한 시크릿으로 못 뜬다, (3) RS256+kid 전환은 발급·검증·키 배포 파이프라인을 바꾸는 큰 작업이라
  **MVP 단계 리스크 대비 비용이 크다**. → "시크릿 관리로 현 위험을 통제 가능하니 RS256은 후속 과제로 명시 보류".
- 나머지 7건(#1~#7: 토큰 URL 노출, localStorage, OAuth state, refresh 회전, XFF, 인증 레이트리밋, CORS)은 완료.

### Q. 보안 회귀 테스트 7건은 각각 무엇을 검증하나
`security-test-scenarios.md`의 시나리오 B~H(7건). 각각:
1. **B (#1·#2)**: 토큰이 URL·localStorage·JS 어디에도 없고 access/refresh 쿠키가 **HttpOnly**, csrf는 비-HttpOnly인지.
2. **C (#3)**: 위조·재사용 OAuth `state`가 401로 거부되는지(랜덤 nonce + `getdel` 1회 소비).
3. **D (CSRF)**: 쿠키 인증 unsafe 요청에 CSRF 헤더 없으면 403, 일치하면 통과. **Bearer(모바일)는 면제**되는지.
4. **E (#4)**: refresh **1회용 회전** + 재사용·로그아웃 후 401(jti 화이트리스트 재사용 탐지), 웹 무중단 자동 재발급.
5. **F (#5·#6)**: 인증 경로 분당 한도(기본 10) 초과 시 429, **XFF 위조로 레이트리밋 우회 불가**.
6. **G (#7)**: 허용 오리진만 CORS 통과, 비허용 오리진은 `Access-Control-Allow-Origin` 미부여.
7. **H (회귀)**: 모바일 플로우(일회용 코드→exchange→Bearer)가 **변경 없이** 동작하는지.
- 별도로 인프라 불필요 **자동 단위 테스트**(`backend/tests/test_security.py`, 시나리오 A): 토큰 라운드트립·위조 거부,
  OAuth URL이 랜덤 state를 싣는지, `_client_ip`의 XFF 처리, `set_auth_cookies`의 쿠키 속성.
- **정직한 한계**: B~H는 **수동 E2E 절차서**(curl/DevTools 체크리스트)이고, CI에서 자동 실행되는 건 A(단위)뿐이다.
  "회귀 테스트"라 부르지만 대부분 사람이 도는 검증이라, 진짜 회귀 방지는 이 시나리오들을 자동화해야 완성된다.

## 8. 테스트

### Q. 실버그 16건 중 가장 발견하기 어려웠던 것과 그 이유
- 후보는 **무음 실패(silent failure)** 부류다. 가장 어려운 건 **§2-5 색인 매핑 이원화**로 본다.
  - **이유**: 예외도, 에러 로그도, 500도 없다. 검색이 **200을 반환하며 그냥 결과가 빈다.** 영어/태그 검색만
    조용히 안 되는데, 한국어 검색은 되니 "검색은 동작한다"고 착각하기 쉽다. 코드를 읽어도 두 매핑이 각각은 유효해
    잘못을 못 느낀다 — **"둘이 다르다"는 사실 자체가 버그**라 단일 파일 리뷰로는 안 잡힌다.
  - 결국 **사용자 시나리오 전수 실UI 검증**(영어 키워드로 실제 검색)에서야 드러났다.
- 경쟁 후보 **§2-2 (204 JSON 파싱)**: **DB에는 저장되는데 UI는 롤백**되는, "되는데 안 되는 것처럼 보이는" 버그.
  낙관적 업데이트가 원복돼 사용자에겐 "안 됨"이지만 서버 로그·DB는 정상이라 원인 지목이 어렵다.
- 공통 교훈: **단위 동작이 아니라 사용자 시나리오 단위**로 봐야 무음 실패류가 잡힌다(`§3-7`).

### Q. Playwright E2E의 flaky 테스트는 어떻게 다뤘나
- **정직한 한계**: 레포에 Playwright 스펙 파일과 flaky 재시도 설정(예: `retries`, `expect` 폴링 표준화)이 체계적으로
  정비돼 있다고 코드로 확증하긴 어렵다. 문서(`§3-7`)는 "Playwright E2E(콘솔 에러 0 기준)"로 검증했다고 기술한다.
- 원칙 수준의 답: E2E flaky의 주범은 (1) 하이드레이션/비동기 상태 레이스, (2) 고정 `sleep`. 이 프로젝트는 실제로
  **하이드레이션 레이스 버그**(`§2-4` admin 가드가 persist 재수화 전 리다이렉트)를 겪고 `onFinishHydration` 대기로 고쳤다 —
  같은 원리로 E2E도 **명시적 상태 대기(요소/네트워크 idle)로 sleep을 대체**하는 게 flaky 억제의 정석이다.
- 즉 "제품 코드의 레이스를 없앤 것"이 E2E 안정성에도 직결됐다는 답이 정직하다. 전용 flaky 격리 리포트/쿼런틴 체계는 미구현.

### Q. "콘솔 에러 0" 기준을 통과 조건으로 삼은 이유는
- 프론트에서 **콘솔 에러는 "조용한 고장"의 대리 지표**다. React key 경고(`§2-6` 태그 `#undefined`), 파싱 예외(`§2-2`),
  하이드레이션 불일치는 화면이 그럴듯해도 콘솔에 먼저 뜬다. 기능 assert만으로는 이 계열을 놓친다.
- 이 프로젝트의 버그 상당수가 "**200인데 UI가 이상**"한 류라, 콘솔 에러 0을 게이트로 두면 **명시적 실패가 없는 결함**을
  기계적으로 걸러낸다. "에러가 없다"가 아니라 "**침묵하는 에러까지 0**"을 요구한 것.
- **정직한 한계**: 콘솔 에러 0은 필요조건이지 충분조건이 아니다. 로직 오류(잘못된 값이지만 에러는 안 남)는 못 잡는다 —
  기능 assert와 병행해야 의미가 있다.

### Q. 만료 JWT로 전 기능 401이 나는 문제는 어디가 원인이었나
- **두 층의 원인이 겹친 복합 버그**다(`§2-1` + `§2-3`):
  1. **프론트(§2-1)**: JWT 60분 만료인데 zustand `persist`가 **만료된 token+user를 그대로 유지**했다. 만료 토큰을
     감지·정리하는 로직이 프론트 어디에도 없어, 헤더는 "로그인" 상태로 보이는데 좋아요·북마크·추천·북마크 페이지의
     모든 인증 요청이 **401로 조용히 롤백** → 사용자엔 "전부 안 됨".
  2. **백엔드(§2-3)**: 공개 피드의 **선택적 인증(`optional_user`)** 이 만료/위조 토큰에 **401을 던졌다.** 그래서
     토큰이 만료되면 로그인 여부와 무관해야 할 **공개 피드 전체까지** 깨졌다.
- **수정**:
  - 프론트: `isTokenExpired(token)`으로 persist 복원 시 만료면 즉시 정리, 401 전역 핸들러로 세션 중 만료 시 자동 logout,
    401이면 조용한 롤백 대신 `AuthModal` 재로그인 유도.
  - 백엔드: 선택 인증 경로는 decode 실패 시 **게스트 폴백**, 필수 인증 경로만 401 유지(`dependencies.py`).
- **한 줄 원인**: "만료 토큰을 (프론트는) 정리하지 않고, (백엔드 선택 인증은) 게스트로 낮추지 않고 401로 올린 것."
  이후 인증은 아예 HttpOnly 쿠키 + refresh 자동 회전으로 개편돼(§7), 만료는 프론트가 `/auth/refresh` 1회 재시도로
  **무중단 처리**한다.

## 9. 답변 준비 우선순위 (요약 카드)

1. **비용 $0.0037 / 게이트 0.85·0.9의 산출 근거**
   - $0.0037 = Haiku 단가 × 실측 토큰, **단일 카드 1건 표본**(평균 아님, 단가는 코드 하드코딩).
   - 0.85(역번역 코사인)·0.9(TF-IDF 제목)는 **통계 튜닝이 아니라 보수적 휴리스틱**. 오탐/미탐 비율 미측정.
     문서가 0.85 vs 0.9로 엇갈리나 **코드가 정답(디듀프 0.9)**.

2. **검색엔진 전환 — 첫 선택의 실패를 어떻게 설명하나**
   - "Meilisearch=MVP 적정선 → 검색 비중을 과소평가 → 재평가로 전환"이 정직한 프레이밍.
   - 그리고 **실제 구현은 nori가 아니라 cjk_bigram(OpenSearch/Bonsai 무료 = 플러그인 불가)** — 문서의 "nori" 서술을
     **정정**하며 트레이드오프(재현율 확보 / 정밀도 포기)를 설명.

3. **인증 개편 전후 (무엇이 취약했고 무엇으로 막았나)**
   - 전: localStorage JWT + URL 토큰 노출(XSS·로그 잔류). 후: HttpOnly 쿠키(XSS로 토큰 불가) +
     SameSite(1차) + 더블 서브밋 CSRF(2차) + refresh jti 회전(재사용 탐지). 모바일은 Bearer 유지.
   - 남긴 것: #8 HS256(수용), 모바일 PKCE(후속).

4. **무음 실패 후 추가한 감지 체계 — 정직한 버전**
   - 색인 매핑 이원화(§2-5)를 겪고 **매핑 단일 출처화**로 원인을 제거했다. 다만 이는 "재발 원인 제거"이지
     **런타임 드리프트 감지기 추가는 아니다**(색인↔DB 대조 잡, 매핑 diff 헬스체크는 미구현 — 후속 과제).
   - 실제로 무음 실패를 잡아낸 것은 **"사용자 시나리오 전수 실UI 검증 + 콘솔 에러 0"** 문화였다.
