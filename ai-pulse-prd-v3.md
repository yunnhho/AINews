## 1. 제품 개요

### 1.1 한 줄 정의

AI/LLM 관련 최신 소식을 **카드 뉴스 형태**로 자동 수집·요약·번역하여, 개발자·디자이너·일반 사용자가 3분 안에 트렌드를 파악할 수 있는 웹/앱 서비스.

### 1.2 해결하려는 문제

AI 분야는 하루에도 수십 건의 모델 출시, 논문, 엔지니어링 기법이 쏟아진다. 현재 이 정보를 소비하는 방식에는 세 가지 핵심 문제가 있다.

첫째, **정보 파편화**다. TechCrunch, arXiv, Hacker News, X, 블로그 등 소스가 흩어져 있어 매일 30분 이상 수동으로 돌아다녀야 핵심을 파악할 수 있다.

둘째, **언어 장벽**이다. 핵심 소식의 80% 이상이 영어로 발행되며, 기존 번역 서비스로는 기술 용어의 맥락을 정확히 전달하기 어렵다.

셋째, **관점 부재**다. 같은 "Claude 4.6 출시" 뉴스라도 개발자는 API 변경사항이, 디자이너는 UI 컴포넌트가, PM은 가격 정책이 궁금하다. 기존 서비스는 이를 구분하지 않는다.

### 1.3 제품 비전

> "AI 소식을 **3분 안에**, **내 관점에 맞게**, **카드 한 장으로** 소비한다."

### 1.4 타겟 사용자

| 페르소나 | 설명 | 핵심 니즈 |
|---|---|---|
| **주니어~미드 개발자** | AI를 활용한 개발에 관심, 최신 기법을 따라가고 싶은 사람 | 코드 예제, API 변경사항, 프레임워크 비교, 벤치마크 |
| **프로덕트 디자이너** | AI 기반 서비스의 UX/UI 트렌드를 파악하고 싶은 사람 | AI 서비스 UI 변화, 디자인 도구 업데이트, AI+디자인 워크플로 |
| **일반 사용자 / PM** | AI 기술의 비즈니스 임팩트를 이해하고 싶은 사람 | 서비스 출시 소식, 시장 동향, 쉬운 용어 설명 |

```

### 2.2 세부 태그 (하위 분류)

대분류 내에서 더 세밀한 필터링을 위해 태그 시스템을 사용한다.

**프로그래밍 태그**: `#LLM` `#ComputerVision` `#RAG` `#FineTuning` `#MCP` `#AgenticAI` `#Prompt-Engineering` `#MLOps` `#OpenSource` `#Benchmark`

**디자인 태그**: `#UI-UX` `#Figma` `#ImageGen` `#VideoGen` `#DesignSystem` `#Prototyping` `#AIxBranding`

**일반 태그**: `#ModelRelease` `#Funding` `#Regulation` `#Ethics` `#Market` `#Tutorial` `#Opinion`

### 2.3 자동 분류 로직

AI가 뉴스 본문을 분석하여 대분류를 판정한다. 판정 기준은 다음과 같다.

| 판정 신호 | 분류 결과 |
|---|---|
| 코드 블록, API endpoint, 벤치마크 수치, GitHub 링크 포함 | 프로그래밍 |
| UI 스크린샷, 디자인 도구명, UX 패턴 언급, 시각적 사례 중심 | 디자인 |
| 위 두 가지에 해당하지 않거나, 비즈니스/정책/시장 중심 내용 | 일반 |
| 복수 카테고리에 해당 | 주요 카테고리 1개 + 보조 카테고리 태그 |

**⚠️ 예상 문제점**: 순수 AI 자동 분류의 정확도는 초기에 80~85% 수준이 될 가능성이 높다. 이를 보완하기 위해 사용자의 "잘못된 분류 신고" 버튼을 두고, 신고 데이터가 일정량 쌓이면 분류 프롬프트를 개선하는 피드백 루프를 설계한다.

## 3. 핵심 기능 요구사항

### 3.1 카드 뉴스 시스템

#### FR-01. 카드 타입 이원화

콘텐츠 성격에 따라 카드를 **두 가지 타입**으로 구분한다. 같은 카테고리(예: 프로그래밍)라도 "Claude 4.7 출시 뉴스"와 "Multi-Agent Harness 패턴"은 사용자가 기대하는 정보 구조가 다르기 때문이다.

| 타입 | 코드 | 설명 | 주요 소스 |
|---|---|---|---|
| **일반 뉴스** | `NEWS` | 모델 출시, 기업 동향, 시장 분석 등 시의성 중심 콘텐츠 | 뉴스 RSS, 기업 공식 블로그 |
| **엔지니어링 기법** | `TECHNIQUE` | RAG 패턴, 에이전트 하네스, MCP 활용법 등 실무 적용 가능한 기법 | GitHub, 엔지니어링 블로그, 뉴스레터, 논문 |

타입별로 카드 템플릿, 수집 파이프라인, AI 프롬프트가 다르게 적용된다.

#### FR-02. NEWS 카드 구성 요소

일반 뉴스 카드는 다음 정보를 포함한다.

```
┌──────────────────────────────────────┐
│  [📰 NEWS]  [카테고리]    [난이도]    │
│                                      │
│  대표 이미지 / 일러스트               │
│                                      │
│  ─────────────────────────────────── │
│  제목 (25자 이내 권장)                │
│  ─────────────────────────────────── │
│                                      │
│  AI 요약 (3~5문장, 한국어)            │
│  • What  무엇이 일어났는가             │
│  • Why   왜 중요한가                  │
│  • Impact 어떤 영향이 있는가           │
│                                      │
│  ─────────────────────────────────── │
│  📰 원본 기사 보기  →  [실제 URL]     │
│  🏷️ #LLM  #ModelRelease              │
│  📅 2026-05-06  ·  출처: TechCrunch  │
│                                      │
│  [❤️ 좋아요]  [🔖 북마크]  [🔗 공유]  │
└──────────────────────────────────────┘
```

#### FR-03. TECHNIQUE 카드 구성 요소

엔지니어링 기법 카드는 **실무 적용 관점**의 구조를 따른다.

```
┌──────────────────────────────────────┐
│  [⚙️ TECHNIQUE]  [카테고리]  [난이도] │
│                                      │
│  대표 이미지 / 다이어그램             │
│                                      │
│  ─────────────────────────────────── │
│  기법 이름 (예: "Multi-Agent Harness")│
│  ─────────────────────────────────── │
│                                      │
│  ▸ 어떤 문제를 해결하나                │
│    (2~3문장으로 상황 설명)             │
│                                      │
│  ▸ 핵심 아이디어                       │
│    (해결 방식 요약, 3~5문장)           │
│                                      │
│  ▸ 코드 예제 / 패턴 (있으면)           │
│    ```python                         │
│    # 핵심 코드 스니펫                  │
│    ```                               │
│                                      │
│  ▸ 적용 시 주의점                     │
│    • 비용 / 성능 / 한계 1~3개          │
│                                      │
│  ─────────────────────────────────── │
│  📚 원본 보기  →  [GitHub/블로그 URL] │
│  🏷️ #Agents  #Harness  #LangGraph    │
│  📅 2026-05-06  ·  출처: LangChain   │
│                                      │
│  [❤️ 좋아요]  [🔖 북마크]  [🔗 공유]  │
└──────────────────────────────────────┘
```

**필수 요소 상세 (공통)**:

- **AI 요약**: 영문 콘텐츠는 한국어로 번역하여 요약. 기술 용어는 업계 관례 따름.
- **원본 URL**: 반드시 원본의 실제 URL 포함 (GitHub Release, 블로그 포스트, 논문 등).
- **카테고리 뱃지**: 프로그래밍(파란색) / 디자인(보라색) / 일반(초록색).
- **타입 뱃지**: NEWS(📰) / TECHNIQUE(⚙️) — 시각적으로 즉시 구분 가능.
- **난이도**: 입문 ⭐ / 중급 ⭐⭐ / 심화 ⭐⭐⭐ (자동 판정).

**⚠️ TECHNIQUE 카드 설계 시 고려한 문제점**:

| 문제 | 대응 |
|---|---|
| 코드 스니펫이 너무 길어 카드 UX를 해칠 수 있음 | 카드에는 최대 15줄 핵심만. 전체 코드는 원본 링크로 유도 |
| 코드 블록 렌더링은 일반 뉴스와 다름 (highlight.js 등 필요) | TECHNIQUE 카드 전용 컴포넌트 분리. Prism.js 또는 Shiki 사용 |
| AI가 코드 예제를 생성하면 오류 가능성 → 원본 코드 인용이 안전 | LLM이 원본에서 코드 추출만 하도록 프롬프트 제약. 코드 생성 금지 |

#### FR-04. 카드 탐색 UX

- **무한 스크롤**: 메인 피드는 카드를 세로로 무한 스크롤
- **카테고리 탭**: 상단에 [전체] [프로그래밍] [디자인] [일반] 탭 고정
- **타입 필터**: [전체] [📰 뉴스] [⚙️ 기법] 토글로 카드 타입 필터
- **태그 필터**: 카테고리 탭 하단에 태그 칩으로 추가 필터링
- **카드 펼치기**: 카드 탭 시 상세 요약 + 원문 링크 확장 표시
- **스와이프 제스처**: 모바일에서 좌/우 스와이프로 북마크/스킵

│     ├─ 소스별: 수집 N / 중복 제거 N / 최종 발행 N                │
│     ├─ 타입별: NEWS N장 / TECHNIQUE N장                          │
│     └─ API 비용: 토큰 N / 비용 $N                                │
└─────────────────────────────────────────────────────────────────┘

예상 배치당 처리량 (확장 후):
  - 수집: 40~60건 → 중복 제거 후: 20~30건 → 발행: 20~30장
  - 그 중 TECHNIQUE 비중: 약 30~40%

예상 일일 발행량: 80~120장 (NEWS 50~70 + TECHNIQUE 30~50)
```

**⚠️ 소스 확장에 따른 트레이드오프**:

| 항목 | 내용 |
|---|---|
| **장점** | 엔지니어링 기법 콘텐츠 확보로 핵심 차별점 강화. 개발자 페르소나 만족도 상승 |
| **단점** | 일일 카드 수 2배 증가 → Claude API 비용 비례 증가 (월 $25~30 → $50~60 예상) |
| **대응** | 배치 내 강력한 중복 필터링 유지. 단순 "출시 알림"성 GitHub Release는 자동 제외 (CHANGELOG 길이 임계값 등) |

#### FR-06. 번역 품질 이중 검증 시스템

영문 뉴스의 한국어 번역 정확도를 보장하기 위해 **2단계 검증**을 적용한다.

**1단계 — 번역 생성**: Claude API가 영문 원문을 한국어로 번역하되, 단순 직역이 아닌 "AI/기술 도메인 맥락을 유지한 의역"을 수행한다. 시스템 프롬프트에 다음 규칙을 포함한다:
- 기술 용어는 업계 관례를 따른다 (예: "fine-tuning"은 "파인튜닝"으로, "inference"는 "추론"으로)
- 고유명사(모델명, 회사명, 라이브러리명)는 원문 그대로 유지한다
- 수치, 벤치마크 결과는 절대 변경하지 않는다

**2단계 — 역번역 검증(Back-Translation Verification)**: 번역된 한국어 텍스트를 다시 영어로 역번역한 뒤, 원문과의 의미 유사도를 측정한다. 역번역은 Claude API로, 유사도 측정은 sentence-transformers(로컬 모델)로 처리하여 비용을 최적화한다.

```
원문(영어) ──→ 번역(한국어)       ──→ 역번역(영어)         ──→ 유사도 비교
            [Claude API]          [Claude API]          [sentence-transformers]
                                                              │
                                                        ≥ 0.85 → 통과
                                                        < 0.85 → 재번역
                                                        3회 실패 → 수동 검토 큐
```

**유사도 측정 방식**: sentence-transformers 라이브러리의 다국어 모델(paraphrase-multilingual-MiniLM-L12-v2)을 사용하여 문장 임베딩 기반 코사인 유사도를 로컬에서 계산한다. API 호출 없이 처리하므로 추가 비용이 발생하지 않는다.

**⚠️ 예상 문제점과 대응**:

| 문제 | 영향 | 대응 방안 |
|---|---|---|
| 역번역에 추가 Claude API 호출 필요 → 비용 증가 | 배치당 추가 비용 발생 | 전체 기사가 아닌 요약문(3~5문장)만 검증하여 토큰 수 최소화. 배치 처리로 호출 수 자체가 줄어든 상태 |
| 기술 용어가 역번역 시 다른 영어 표현으로 바뀜 | 유사도 점수 왜곡 | 기술 용어 사전(glossary)을 관리하여 번역/역번역 시 일관성 유지 |
| 유사도 임계값(0.85) 설정의 적절성 | 너무 높으면 대부분 재번역, 너무 낮으면 오역 통과 | 초기 1개월간 임계값별 통과율을 로깅하고, 수동 검수 결과와 대조하여 보정 |
| sentence-transformers 모델 로딩에 ~500MB 메모리 필요 | 워커 OOM(메모리 부족) | 워커 컨테이너 최소 2GB RAM 확보. 배치 완료 후 모델 언로드 |

#### FR-07. AI 요약 품질 기준

**NEWS 카드 요약 기준**:

- **사실 정확성**: 원문에 없는 내용을 생성하지 않는다 (Hallucination 방지)
- **핵심 보존**: 원문의 What(무엇이), Why(왜 중요한지), Impact(어떤 영향이 있는지)를 반드시 포함
- **길이 제한**: 요약은 3~5문장 (150~300자), 핵심 포인트는 2~3개 불릿
- **톤 일관성**: 카테고리별로 톤을 미세 조정
  - 프로그래밍: 정확하고 기술적인 톤. 구체적 수치/버전 명시
  - 디자인: 시각적 변화를 묘사하는 톤. 비교 표현 활용
  - 일반: 비전문가도 이해할 수 있는 친근한 톤

**TECHNIQUE 카드 요약 기준**:

- **구조 일관성**: 반드시 "문제 → 핵심 아이디어 → 코드(있으면) → 주의점" 4단 구조 유지
- **코드 신뢰성**: 코드는 **원본에서 추출만** 한다. LLM이 새로 생성하지 않는다. 추출 시 원문 코드의 줄 단위 인용을 유지하고, 출처(파일명/링크)를 표시
- **실용성 강조**: "언제 사용하면 좋은가" / "언제 사용하면 안 되는가"를 명시
- **선행 지식 표기**: 이 기법을 이해하려면 알아야 할 개념(예: "LangGraph 기본 사용법 숙지 필요")을 난이도와 연계하여 표시
- **길이 제한**: 각 섹션 2~4문장, 코드는 최대 15줄

### 3.3 사용자 기능

#### FR-08. 좋아요 / 북마크

- **좋아요(❤️)**: 카드에 좋아요를 누를 수 있다. 좋아요 수는 카드에 표시된다.
- **북마크(🔖)**: 카드를 저장하여 "내 북마크" 페이지에서 모아볼 수 있다.
- 두 기능 모두 로그인 필수. 비로그인 사용자에게는 로그인 유도 모달을 표시한다.
- 북마크한 카드는 카테고리별로 분류하여 볼 수 있다.

#### FR-09. 북마크/좋아요 기반 추천

사용자의 좋아요와 북마크 이력을 분석하여 개인화 추천을 제공한다.

**추천 로직 (단계적 발전)**:

- **Phase 1 (MVP)**: 사용자가 좋아요/북마크한 카드의 카테고리·태그·**카드 타입** 빈도를 집계하여, 동일 패턴의 최신 카드를 우선 노출한다. (규칙 기반, 별도 ML 모델 불필요)
  - 예: 사용자가 TECHNIQUE 카드를 70% 비율로 좋아요했다면, 추천 피드도 TECHNIQUE 비중을 높임
- **Phase 2 (v2.0)**: 협업 필터링(Collaborative Filtering) 도입. "나와 비슷한 취향의 사용자가 좋아한 카드"를 추천한다. (사용자 수 1만 명 이상 확보 후 적용)

**⚠️ 예상 문제점**: 초기 사용자 수가 적으면 추천 품질이 낮아지는 콜드 스타트 문제가 발생한다. Phase 1에서 규칙 기반으로 시작하는 이유가 여기에 있다. 또한 "필터 버블" 문제(관심사만 보여주는 현상)를 방지하기 위해, 추천 피드의 20%는 사용자가 아직 접하지 않은 카테고리·타입의 인기 카드를 포함한다.

#### FR-10. 사용자 인증

- **소셜 로그인**: Google, GitHub, Kakao 지원 (자체 회원가입 없음)
- **게스트 모드**: 로그인 없이 카드 열람 가능. 좋아요/북마크/추천은 로그인 필요.

#### FR-11. 공유 기능

- 카드별 고유 URL 생성 (SEO 친화적, `aipulse.kr/cards/{id}`)

- 공유 시 카드의 OG(Open Graph) 이미지 자동 생성

### 3.4 관리자 기능

#### FR-12. 관리 대시보드

완전 자동 수집 방식이므로, 관리 대시보드에서 시스템 상태를 모니터링한다.

- **배치 실행 현황**: 최근 배치 실행 시각, 소스 그룹별(A/B/C/D) 수집 건수, 중복 제거율, 실패 건수

- **번역 품질**: 역번역 검증 통과율, 재번역 횟수 분포, 수동 검토 큐 대기 건수
- **분류 정확도**: 카테고리별 발행 건수, 카드 타입 자동 판정 정확도, 사용자 오분류 신고 건수

- **사용자 지표**: DAU/MAU, 좋아요/북마크 수, 카테고리×타입별 인기 태그
- **비용 모니터링**: Claude API 토큰 사용량, 배치당/일일/월간 비용 추이

## 5. 기술 스택 (확정: 스택 B — Python 백엔드 + React 프론트)

AI 파이프라인의 유연성과 Python ML 생태계의 이점을 중심으로 선정된 스택이다.

### 5.1 스택 구성

| 계층 | 기술 | 선정 이유 |
|---|---|---|
| **프론트엔드(웹)** | Next.js 15 (App Router) | SSR/SSG로 SEO 해결. React 생태계 활용 |
| **모바일 앱** | React Native (Expo) | 웹과 컴포넌트 로직 공유. Expo로 빌드 파이프라인 간소화 |
| **백엔드 API** | FastAPI (Python 3.12) | 비동기 지원. AI 라이브러리와 자연스러운 통합. 자동 API 문서 생성 |
| **데이터베이스** | PostgreSQL (AWS RDS 또는 Neon) | 안정성 검증된 관계형 DB. JSON 타입 지원으로 유연한 스키마 |
| **캐시** | Redis (Upstash 또는 ElastiCache) | 피드 캐싱, 추천 결과 캐싱, 배치 상태 관리 |
| **뉴스 수집 파이프라인** | Celery + Celery Beat + Redis | Python 표준 비동기 작업 큐. Celery Beat으로 6시간 간격 스케줄링 |
| **AI 요약/번역/분류** | Claude API (Anthropic) + LangChain | LangChain으로 프롬프트 체이닝(요약→번역→분류) 구성. 한국어 품질 우수 |

| **검색** | Elasticsearch (또는 Meilisearch) | 한국어 형태소 분석기(nori) 지원. 태그/카테고리 복합 검색 |
| **배포** | AWS ECS Fargate (또는 Railway) | 컨테이너 기반 배포. API 서버와 배치 워커 독립 스케일링 |
| **모니터링** | Sentry + Grafana + Prometheus | 에러 추적 + 배치 파이프라인 상태 + API 성능 종합 모니터링 |
| **CI/CD** | GitHub Actions | PR 기반 자동 테스트 + 배포. ECR 푸시 자동화 |

### 5.2 스택 선정 근거

**이 스택을 선택한 핵심 이유**: Python은 AI/ML 라이브러리(sentence-transformers, LangChain, tiktoken 등)와의 호환이 가장 좋다. 특히 번역 검증에 사용하는 sentence-transformers를 로컬에서 실행하면 임베딩 API 비용을 절감할 수 있다. Celery Beat의 cron 스케줄링이 6시간 배치 패턴과 자연스럽게 맞으며, 배치 작업의 재시도/에러 핸들링이 성숙하다.

**⚠️ 스택 B의 주의할 점**:

| 주의사항 | 상세 | 대응 |
|---|---|---|
| 프론트(TS)와 백엔드(Python) 2개 언어 사용 | 컨텍스트 스위칭 비용 | API 인터페이스를 OpenAPI 스펙으로 명확히 정의. FastAPI 자동 문서화 활용 |
| Celery 워커 메모리 누수 | 장시간 실행 시 OOM 발생 | `--max-tasks-per-child=50` 설정으로 워커 주기적 재시작 |
| sentence-transformers 모델 로딩 (~500MB) | 워커 cold start 시 지연, 메모리 부족 | 워커 컨테이너 최소 2GB RAM. 배치 시작 시 사전 로딩 |
| Elasticsearch 운영 복잡도 | 클러스터 관리, 인덱스 튜닝 | MVP에서는 Meilisearch(운영 간편)로 시작 후, 규모 커지면 Elasticsearch로 전환 가능 |

### 7.3 카드 타입별 필드 활용

| 필드 | NEWS | TECHNIQUE |
|---|---|---|
| `key_points[]` | ✅ 사용 (2~3개 불릿) | ❌ |

| `idea` | ❌ | ✅ 필수 |
| `code_snippet` | ❌ | ⭕ 선택 (있으면 표시) |
| `caveats[]` | ❌ | ✅ 필수 (1~3개) |
| `prerequisites` | ❌ | ⭕ 선택 |

**⚠️ 스키마 설계 시 고려한 문제점**: 카드 타입별로 필드 구조가 크게 다르므로 두 가지 설계안이 가능하다. (1) 단일 테이블에 nullable 컬럼으로 처리, (2) `cards_news`와 `cards_technique`로 테이블 분리. v3.0에서는 **단일 테이블 + 타입별 nullable**을 채택했다. 이유는 (a) 피드 조회 시 두 타입을 섞어서 보여줘야 하므로 JOIN/UNION 비용이 큰 분리 방식은 불리하고, (b) PostgreSQL의 `CHECK` 제약으로 타입별 필수 필드를 강제할 수 있어 데이터 무결성을 보장할 수 있기 때문이다. 스키마 예시:

```sql
ALTER TABLE cards ADD CONSTRAINT card_type_fields CHECK (
  (card_type = 'NEWS' AND key_points IS NOT NULL AND problem IS NULL)
  OR
  (card_type = 'TECHNIQUE' AND problem IS NOT NULL AND idea IS NOT NULL)
);
```

| 이메일 본문에 트래킹 픽셀, 광고 링크가 섞임 | HTML 파싱 시 `<img>`, `utm_` 파라미터 자동 제거 |
| 한 이메일에 여러 콘텐츠가 묶여 있음 (TLDR AI 등) | LLM에게 "이 이메일을 콘텐츠 단위로 분할" 요청 → 각각 별도 카드로 처리 |
| 뉴스레터 자체의 저작권 | 이메일 본문 전체 인용 금지. 링크된 원본 소스를 따라가 그곳의 콘텐츠를 요약. 원본 URL은 반드시 노출 |
| 이메일 계정 잠금/해킹 위험 | 전용 계정 사용. 2FA 필수. 앱 비밀번호로 IMAP 접근 |

### 8.5 소스 그룹별 우선순위 (단계적 도입)

한 번에 모든 소스를 도입하면 파이프라인 복잡도와 노이즈가 폭증한다. **단계별 도입 전략**:

```
┌─────────────────┬─────────────────────────────────────────────┐
│ Phase 1 (MVP)   │ 그룹 A 전체 + 그룹 B-1 (Releases) +          │
│                 │ 그룹 C-1 (기업 블로그 5개) + 그룹 D-1        │
│                 │ (Substack 2개) → 검증된 신호만               │
├─────────────────┼─────────────────────────────────────────────┤
│ Phase 2         │ 그룹 B-2 (Trending) + B-3 (Awesome) +        │
│                 │ 그룹 C-2 (개인 블로그) → 다양성 확보         │
├─────────────────┼─────────────────────────────────────────────┤
│ Phase 3         │ 그룹 D-2 (이메일 뉴스레터)                   │
│                 │ → IMAP 파싱 인프라 구축 필요                 │
└─────────────────┴─────────────────────────────────────────────┘
```

**⚠️ 소스 확장 시 공통 예상 문제점**: 웹 크롤링 방식은 대상 사이트 구조 변경 시 수집이 깨진다. 이를 대비하여 (a) RSS가 있는 소스는 RSS를 우선 사용, (b) 각 소스별 헬스체크 테이블(`source_health`)을 운영하여 연속 실패 시 관리 대시보드에 경고를 표시, (c) robots.txt를 준수하며, (d) 6시간 간격 배치이므로 대상 서버에 부하를 줄 우려가 거의 없다.

### 9.1 Phase 1: 규칙 기반 추천 (MVP)

```
사용자의 최근 30일 좋아요/북마크 이력
        │
        ▼
카테고리·태그 빈도 집계

## 13. 부록: 용어 정의

| 용어 | 설명 |
|---|---|
| **카드 뉴스** | 하나의 AI 관련 콘텐츠를 시각적 카드 형태로 요약한 콘텐츠 단위 |
| **NEWS 카드** | 시의성 중심의 일반 뉴스 카드. What/Why/Impact 3단 구조 |
| **TECHNIQUE 카드** | 엔지니어링 기법 카드. 문제/핵심 아이디어/코드/주의점 4단 구조 |
| **소스 그룹** | 수집 소스의 분류 단위. A(뉴스 RSS), B(GitHub), C(엔지니어링 블로그), D(뉴스레터) |
| **배치(Batch)** | 일정 시간 간격으로 모아서 한 번에 처리하는 방식. 본 서비스에서는 6시간 간격(하루 4회) |
| **하네스(Harness)** | LLM/에이전트를 실제 환경에서 동작시키기 위한 코드 골격. 예: 에이전트 평가 하네스, 멀티 에이전트 하네스 |
| **에이전트 팀(Agent Team)** | 여러 AI 에이전트가 역할을 나누어 협업하는 패턴. CrewAI, AutoGen 등에서 구현 |
| **MCP (Model Context Protocol)** | LLM이 외부 도구·데이터 소스와 표준화된 방식으로 연결되도록 하는 프로토콜 |
| **Awesome 리스트** | 특정 주제의 좋은 자료를 모아둔 GitHub 큐레이션 리포지토리. 새 항목 추가가 곧 새 기법의 등장 신호 |
| **역번역 검증** | 번역된 텍스트를 원래 언어로 다시 번역하여 의미 보존 여부를 확인하는 품질 검증 방법 |
| **코사인 유사도** | 두 벡터 간의 각도를 측정하여 텍스트의 의미적 유사성을 0~1 사이 값으로 표현하는 지표 |
| **콜드 스타트** | 사용자 데이터가 부족하여 추천 시스템이 제대로 작동하지 않는 초기 상태 |
| **필터 버블** | 추천 알고리즘이 사용자의 기존 관심사만 반복 노출하여 시야가 좁아지는 현상 |
| **TF-IDF** | Term Frequency-Inverse Document Frequency. 문서 내 단어의 중요도를 수치화하는 방법. 중복 기사 탐지에 활용 |
| **OG 이미지** | SNS 공유 시 미리보기에 표시되는 Open Graph 프로토콜 기반 대표 이미지 |
| **nori** | Elasticsearch에서 한국어 형태소 분석을 담당하는 플러그인 |
| **SSR/SSG** | Server-Side Rendering / Static Site Generation. SEO와 초기 로딩 성능을 위한 렌더링 방식 |
| **Celery Beat** | Celery의 주기적 작업 스케줄러. cron 표현식으로 배치 실행 시각을 정의 |
| **ECS Scheduled Task** | AWS ECS에서 특정 시간에만 컨테이너를 기동하여 작업을 수행하는 방식. 상시 가동 대비 비용 절감 |
| **IMAP** | Internet Message Access Protocol. 이메일 서버에서 메일을 읽어오는 표준 프로토콜. 뉴스레터 자동 수집에 활용 |
| **GitHub Releases API** | GitHub 리포지토리의 릴리스(버전 발행) 정보를 조회하는 API. 프레임워크 업데이트 추적의 핵심 |
