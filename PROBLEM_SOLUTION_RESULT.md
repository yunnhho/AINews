# AI Pulse — 문제 · 해결 · 결과 (PSR)

> AI/LLM 뉴스 카드 플랫폼을 구축하며 마주친 핵심 기술 의사결정을
> **문제 → 해결 → 결과** 형식으로 정리한 문서.
>
> **수치 표기 규칙**
> - `[측정]` : 코드/벤치마크로 실제 확인한 값
> - `[설계목표]` : 임계값·파라미터 등 구현에 반영된 설계 기준값
> - `[예상]` : 운영 트래픽 부재로 산출한 추정치 (실데이터 검증 전)

## 1. 한국어 검색 품질 — Meilisearch → Elasticsearch(nori)

### 문제
- 초기 검색 엔진으로 Meilisearch를 도입했으나, 한국어 형태소 단위 검색이 약했다.
- `"RAG 패턴"` 검색 시 `"RAG"`, `"패턴"` 토큰이 조사·어미와 분리되지 않아
  연관 문서 누락이 발생. 영문 위주 토크나이저로는 한국어 복합어 매칭이 불안정.
- 카드 요약·문제·아이디어 필드가 **한국어 번역문**이라 검색 품질이 곧 서비스 품질.

### 해결
- 검색 엔진을 **Elasticsearch 8.17 + nori 형태소 분석기**로 전환 (P6-2 / Session 8).
- `nori_tokenizer` + `nori_part_of_speech` 필터로 한국어 커스텀 analyzer 구성.
- 검색 필드(`title`/`summary`/`problem`/`idea`)는 `korean` analyzer,
  필터 필드(`category`/`card_type`/`difficulty`/`tags`)는 `keyword`로 분리 매핑.
- 카드 INSERT 시 자동 인덱싱(`tasks/index_card.py`), `setup_index()`로 멱등 초기화.

```
backend/app/services/search.py   — AsyncElasticsearch 래퍼 + nori 매핑
backend/es/setup.py              — 인덱스 생성 + 전체 재인덱싱
```

### 결과
| 항목 | 이전(Meilisearch) | 이후(ES + nori) |
|---|---|---|
| 한국어 형태소 분석 | 미지원(공백/단순 토큰) | nori 형태소 단위 색인 `[설계목표]` |
| 복합 필터(카테고리+타입+난이도) | 부분 지원 | term 필터 조합 완전 지원 `[측정]` |
| `"RAG"` → `"RAG 패턴"` 연관 매칭 | 누락 | 형태소 매칭으로 재현 `[설계목표]` |
| 동의어/품사 필터 확장성 | 제한적 | analyzer 체인으로 확장 가능 `[측정]` |

> 한국어 재현율(recall) 개선 효과 `[예상]`. 실서비스 쿼리 로그 확보 후
> Meilisearch 대비 A/B 검색 품질 정량 비교 예정.

## 2. 추천 정확도 — 규칙 기반 → 협업 필터링(ALS)

### 문제
- MVP 초기 추천은 규칙 기반(P1-4): 최근 30일 좋아요·북마크 이력의
  카테고리·태그 빈도를 집계해 `80% 상위 패턴 + 20% 미접촉 카테고리`로 구성.
- 빈도 기반이라 **사용자 간 취향 유사성**을 활용하지 못함 →
  "나와 비슷한 사람이 좋아한 카드" 발굴 불가, 추천 다양성이 빈도에 갇힘.

### 해결
- 협업 필터링(P6-1 / Session 7)으로 업그레이드, **규칙 기반은 폴백으로 유지**.
- 사용자-카드 상호작용 매트릭스 가중치: `like=1.0`, `bookmark=2.0`, `view=0.3`.
- `implicit` 라이브러리 **ALS**(`factors=64, iterations=20, regularization=0.1`)
  일 1회 재학습 → user/item 팩터를 Redis에 직렬화 저장.
- 조회 시 `item_factors @ user_vec` 코사인 Top-K. 모델에 없는 신규 사용자나
  상호작용 < 5건이면 규칙 기반으로 자동 폴백(cold start 대응).

```
pipeline/tasks/train_cf.py            — ALS 학습 + Redis 저장
backend/app/services/recommendations.py — CF Top-K + 규칙 기반 폴백
```

### 결과
| 항목 | 이전(규칙 기반) | 이후(ALS + 폴백) |
|---|---|---|
| 추천 근거 | 본인 이력 빈도만 | 유사 사용자 잠재요인(64차원) `[설계목표]` |
| Cold start 처리 | 해당 없음(항상 규칙) | < 5 상호작용 → 규칙 폴백 `[측정]` |
| 모델 갱신 | 요청 시 즉시 집계 | 일 1회 사전 학습(조회 지연 ↓) `[설계목표]` |
| 필터 버블 | 20% 강제 다양성 | 잠재요인 기반 + 폴백 다양성 `[예상]` |

> 좋아요/북마크/뷰 로그 누적 후 추천 CTR·정밀도로 규칙 기반 대비
> 정량 개선폭 측정 예정 `[예상]`.

## 3. 번역 신뢰성 — 역번역 코사인 검증

### 문제
- 영문 원문을 Claude로 한국어 요약·번역하는데, LLM 번역은
  **의미 왜곡·환각**이 간헐적으로 발생할 수 있음.
- 잘못된 번역이 그대로 카드로 발행되면 신뢰도 직결 → 자동 품질 게이트 필요.

### 해결
- **역번역(back-translation) + 코사인 유사도** 검증 도입.
- KO 요약을 다시 EN으로 역번역 → 원문과 임베딩 코사인 유사도 측정.
- **임계값 0.85 미만이면 발행 차단** → `manual_review` 큐로 이동, Admin에서 수동 검토.
- 통과 실패 시 최대 재시도, 영어 원본 카드는 `is_published=False` 초안 저장.

```
pipeline/pipeline/ai/backtranslate.py  — _SIM_THRESHOLD = 0.85
pipeline/pipeline/ai/publisher.py      — 미통과 카드 초안 처리
```

### 결과
| 항목 | 이전(검증 없음 가정) | 이후(역번역 게이트) |
|---|---|---|
| 번역 품질 보증 | 무검증 발행 | 코사인 ≥ 0.85만 자동 발행 `[설계목표]` |
| 저품질 번역 처리 | 사용자 노출 | manual_review 격리 `[측정]` |
| 운영 가시성 | 없음 | Admin 번역 통과율 KPI 노출 `[측정]` |

> 경보 기준: 번역 통과율 < 70%(최근 100건)이면 일 1회 알림 발송 `[설계목표]`.

## 4. 중복 카드 — 멀티소스 수집 디듀프

### 문제
- 소스 그룹 A(RSS)·B(GitHub)·C(블로그)·D(뉴스레터) 동시 수집 →
  **같은 뉴스가 여러 소스에서 중복 유입**.
- 단순 URL 비교만으로는 제목만 같고 URL이 다른 재게시·미러를 못 거름.

### 해결
- 2단계 디듀프(`pipeline/filters/dedup.py`):
  1. **URL SHA-256 해시** 완전 일치 제거.
  2. **TF-IDF 타이틀 유사도** — `char_wb` 분석기 `ngram(2,3)`로
     코사인 유사도 **≥ 0.85** 쌍 제거. 충돌 시 **content가 더 긴 카드 유지**.
- DB 기존 카드와의 중복도 별도 필터링. AI 처리 **이전 단계**라 토큰 낭비 차단.

### 결과
| 항목 | 이전(URL만) | 이후(URL + TF-IDF) |
|---|---|---|
| 동일 URL 중복 | 제거 | 제거 `[측정]` |
| 제목 유사 재게시 | 통과(중복 노출) | 0.85 유사도로 제거 `[설계목표]` |
| 중복 시 보존 기준 | 임의 | 더 긴 본문 우선 `[측정]` |
| AI 비용 영향 | 중복분까지 호출 | 중복 제거 후 호출(비용 ↓) `[예상]` |

> char n-gram 기반이라 한국어/영어 제목 모두에서 띄어쓰기·조사 차이에 강건.

## 5. 피드 응답 지연 — Redis 캐시

### 문제
- 메인 피드는 카테고리·타입·태그·난이도·커서 조합으로 매 요청 DB 조회.
- 동일 조건 반복 요청이 많은 피드 특성상 DB 부하·응답 지연 우려.

### 해결
- 피드 응답을 **Redis 캐시(TTL 5분)**.
- 캐시 키: `feed:{category}:{card_type}:{tags}:{difficulty}:{cursor}:{limit}`
  — 조합별 분리 캐싱으로 정확도 유지.

### 결과
| 항목 | 이전(매번 DB) | 이후(Redis 5분) |
|---|---|---|
| 반복 조회 응답원 | PostgreSQL | Redis 메모리 히트 `[설계목표]` |
| DB 부하 | 요청 비례 | 캐시 미스에만 발생 `[예상]` |
| 신선도 | 즉시 | 최대 5분 지연(배치 6시간 주기 대비 무영향) `[측정]` |

> 배치가 KST 00/06/12/18시 4회라 5분 TTL은 신선도 손실이 사실상 없음.

## 6. 소스 안정성 — 헬스 추적 + 자동 비활성화 + 경보

### 문제
- 외부 RSS/스크래핑/IMAP 소스는 **언제든 죽거나 구조가 바뀜**.
- 죽은 소스를 매 배치 재시도하면 자원 낭비 + 장애 인지 지연.

### 해결
- `source_health`로 소스별 연속 실패 추적.
- **연속 실패 3회 이상 → Slack + 이메일 즉시 경보** (동일 소스 1시간 쿨다운).
- 404 등 영구 실패는 **자동 비활성화**(`enabled=False`)로 재시도 차단.
- 배치 전체 수집 0건도 즉시 경보.

```
pipeline/models/source_health.py       — 실패 카운트 + 자동 비활성화
backend/app/services/alerting.py       — Slack/SMTP + 쿨다운
pipeline/tasks/check_source_health.py  — 배치 후 상태 점검
```

### 결과
| 항목 | 이전(무감시 가정) | 이후(헬스 + 경보) |
|---|---|---|
| 장애 인지 | 수동 발견 | 연속 3회 실패 시 자동 경보 `[설계목표]` |
| 죽은 소스 재시도 | 매 배치 반복 | 자동 비활성화로 차단 `[측정]` |
| 경보 노이즈 | — | 1시간 쿨다운으로 중복 억제 `[설계목표]` |

## 7. AI 호출 비용 — 타입별 프롬프트 + 입력 절단

### 문제
- Claude API 호출이 파이프라인 최대 비용원. 무절제한 호출은 비용 폭증.
- 긴 원문 전체를 프롬프트에 넣으면 입력 토큰이 선형 증가.

### 해결
- **카드 타입별 프롬프트 분리**(NEWS / TECHNIQUE) — 불필요한 지시 토큰 제거.
- 원문은 **`content[:3000]`로 절단** 후 요약 → 입력 토큰 상한 고정.
- 디듀프를 AI 단계 **앞단**에 배치(§4) → 중복분 호출 원천 차단.
- 코드 스니펫은 LLM 생성 금지, **원본에서 추출만** → 생성 토큰 절감 + 정확성.
- Admin 비용 대시보드(`/admin/costs/daily`)로 일별 토큰·USD + 예산 게이지 노출.

### 결과
| 항목 | 이전(소박한 방식) | 이후(분리+절단+선디듀프) |
|---|---|---|
| 입력 토큰/카드 | 원문 전체 비례 | 최대 3000자 상한 `[설계목표]` |
| 중복 카드 AI 호출 | 발생 | 디듀프 후 0 `[측정]` |
| 코드 스니펫 토큰 | 생성 비용 발생 | 추출(생성 0) `[측정]` |
| 비용 가시성 | 없음 | 일별 KPI + 예산 경보 `[측정]` |

## 6. 면접 예상 Q&A (방어 포인트)

- **Q. LLM 번역 품질은 어떻게 보장했나?**
  → 한국어 요약을 다시 영어로 역번역해 원문과 임베딩 코사인 유사도를 측정,
  0.85 미만이면 발행을 막고 `manual_review` 큐로 보내 사람이 검토. 재시도 로직 포함.
- **Q. 왜 Meilisearch에서 Elasticsearch로 바꿨나?**
  → 한국어 형태소 검색이 핵심인데 Meilisearch 토큰화가 조사·복합어에 약했다.
  ES의 nori 분석기로 형태소 단위 색인 + 복합 필터를 얻기 위해 전환.
- **Q. 추천은 왜 규칙 기반에서 CF로 바꿨고, cold start는?**
  → 규칙 기반은 본인 이력 빈도만 봐 유사 사용자 취향을 못 쓴다. ALS로 잠재요인을
  학습하되, 모델에 없거나 상호작용 5건 미만이면 규칙 기반으로 폴백.
- **Q. 비용은 어떻게 관리했나?**
  → 타입별 프롬프트 분리 + 입력 3,000자 절단 + AI 호출 전 디듀프로 토큰을 줄이고,
  코드 스니펫은 생성 대신 추출. 일별 토큰·비용을 관리자 대시보드로 가시화.
- **Q. 단일 cards 테이블에 두 카드 타입을 넣은 이유는?**
  → 타입별 nullable 컬럼 구조로 공통 쿼리·인덱스·캐시를 단순화. 타입 분기는
  `card_type` 컬럼 + 애플리케이션 레이어에서 처리.

> ⚠️ **면접 정직성**: 본 프로젝트는 MVP 개발 단계로 실사용자 트래픽 지표는 없음.
> "검색 재현율 X% 향상" 같은 **운영 수치는 주장하지 말 것**. 대신 "역번역 0.85 게이트
> 설계", "nori 형태소 전환", "디듀프로 AI 호출 0" 등 **구현·설계 사실**로 답할 것.
